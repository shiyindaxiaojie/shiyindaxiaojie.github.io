Java 集合

  1. 集合有哪些，继承关系，原理？
    集合分 2 个顶层接口，Iterable（有序）和 Map（无序）
    顺序表：ArrayList（数组）、Vector（线程安全）-> AbstractList -> List -> Collection -> Iterable，ArrayList 初始容量为 10，如果添加元素后超过设定的容量大小，自动扩容为原来的 1.5 倍，而 Vector 是 2 倍
    链表：LinkedList（内建对象 Node 包含 next、prev，并保存 first、last） -> AbstractList -> List -> Collection -> Iterable，因为保存了 first、last，可以代替 Stack 的实现
    哈希散列：HashMap（内建对象 Entry 包含 key、value，使用 Entry 数组保存，通过 hash(key) 获取数组的下标）-> AbstractMap -> Map，HashMap
    哈希表：Hashtable（和 HashMap 类似，但每个方法都加了 synchironized）-> Dictionary && Map
    初始容量为 16，为了避免 hash 碰撞，同一个 hash 值的数组元素用链表保存，当相同数超过 8，在 JDK8 会自动转换为红黑树，小于 6 时退化为链表
    哈希集合：HashSet（使用 HashMap 保存，key 为元素，value 为一个内置的 Object）-> AbstractSet && Set -> Collection -> Iterable
    有序散列：TreeMap（Comparator 比较 Key，用到红黑树）-> SortedMap -> Map
    有序集合：TreeSet（Comparator 比较元素）-> SortedTree -> Set -> Collection -> Iterable

Java I/O

  1. BIO、NIO、AIO 区别
    BIO：用户进程发起一个 I/O 操作后，必须等待 I/O 操作的完成，只有当真正完成了 I/O 操作以后，用户进程才能运行。
    NIO：用户进程发起一个 I/O 操作以后便可返回做其他事情，但是用户进程需要时不时地询问 I/O 操作是否就绪。
    AIO：用户进程发起一个 I/O 操作以后，不等待内核 I/O 操作的完成，内核完成 I/O 操作以后会通知用户进程。
    同步和异步是针对应用程序和内核的交互而言的。阻塞和非阻塞是针对于进程在访问数据时，根据 I/O 操作的就绪状态来采取的不同方式，说白了是一种读取或写入操作函数的实现方式。阻塞方式下，读取或写入函数将一直等待，而非阻塞方式下，读取或写入函数会立即返回一个状态值。
  2. I/O 模型
    每个请求都需要独立的线程完成数据 Read，业务处理，数据 Write 的完整操作问题。
    当并发数较大时，需要创建大量线程来处理连接，系统资源占用较大。
    连接建立后，如果当前线程暂时没有数据可读，则线程就阻塞在 Read 操作上，造成线程资源浪费。
  3. I/O 复用模型
    在 I/O 复用模型中，会用到 Select，这个函数也会使进程阻塞，但是和阻塞 I/O 所不同的是这两个函数可以同时阻塞多个 I/O 操作。而且可以同时对多个读操作，多个写操作的 I/O 函数进行检测，直到有数据可读或可写时，才真正调用 I/O 操作函数。

Java 多线程

  1. 进程和线程的区别，进程间如何通讯，线程间如何通讯
    一个程序至少有一个进程，一个进程至少有一个线程。线程是进程的一个实体，是 CPU 调度和分配的基本单位，是比进程更小的能独立运行的基本单位。进程在执行过程中拥有独立的内存单元，而多个线程共享内存。线程有自己的堆栈和局部变量，与同属一个进程的其他的线程共享进程所拥有的全部资源，但线程之间没有单独的地址空间，一个线程死掉就等于整个进程死掉，所以多进程的程序要比多线程的程序健壮，但在进程切换时，耗费资源较大，效率要差一些。一个线程可以创建和撤销另一个线程，同一个进程中的多个线程之间可以并发执行。
    进程通讯：管道、信号量、共享内存、消息队列、套接字 Socket
    线程通讯：锁机制（互斥锁、条件变量、读写锁）
  2. 线程状态有哪些，如何切换？
    线程状态分为初始、运行（运行中+就绪）、阻塞、等待、超时等待、终止
    初始（NEW）：新创建了一个线程对象，但还没有调用 start() 方法。
    运行（RUNNABLE）：Java线程中将运行中（RUNNING）和就绪（READY）两种状态笼统的称为“运行”。线程对象创建后，其他线程（比如 main 线程）调用了该对象的 start() 方法。该状态的线程位于可运行线程池中，等待被线程调度选中，获取 CPU 的使用权，此时处于就绪状态。就绪状态的线程在获得 CPU 时间片后变为运行中状态。
    阻塞（BLOCKED）：表示线程可能是因为同步操作等待，阻塞于锁。
    等待（WAITING）：进入该状态的线程需要等待其他线程做出一些特定动作（通知或中断）。
    超时等待（TIMED_WAITING）：该状态不同于 WAITING，它可以在指定的时间后自行返回。
    终止（TERMINATED）：表示该线程已经执行完毕。
  3. 线程中的 wait()、sleep()、yield() 的区别
    wait 方法是 Object 的一部分，而 sleep 和 join 是 Thread 类的一部分。
    wait 方法主要用于线程间通信，释放锁给其他线程使用，调用 wait 方法时会获得阻塞锁，再进入就绪状态，可以被 notify 和 notifyAll 唤醒
    sleep 方法只是短时间内暂停当前线程，不释放锁，调用 sleep 方法时立即进入就绪状态，不能被 notify 和 notifyAll 唤醒
    yield 方法不保证当前的线程会暂停或者停止，但是可以保证当前线程在调用 yield 方法时会放弃 CPU

Redis

  1. Redis 数据类型有哪些？
    String：字符串，简单缓存、并发技术
    Hash：散列，ZipList，HashTable，rehash概念、提高负载因子，符合多重 key的情况，比如用户的购物车
    List：列表，ZipList，LinkedList，双向队列，可以用作消息队列，可以先进先出，也可以先进后出
    Set：集合，intSet，HashTable，交并合集
    ZSet：有序集合，ZipList，SkipList
    JSON：Redis 4.0 版本增加 rejson，支持 JSON 格式存储

  2. Redis 应用场景
    缓存存取
    利用 List 的先进先出实现消息队列
    分布式锁
    共享 Session
    通过 ZSet 的分值排序，如评论排行榜

  4. Redis 数据过期策略
    被动删除：当读/写一个已经过期的 key 时，会触发惰性删除策略，直接删除掉这个过期 key
    主动删除：由于惰性删除策略无法保证冷数据被及时删掉，Redis 会定期主动淘汰一批已过期的 key
    超过内存限制被清除：当前已用内存超过 maxmemory 限定时，触发主动清理策略

  6. 如何解决 Redis 缓存雪崩、穿透、击穿
    缓存雪崩：随机过期时间、永不过期
    缓存穿透：表示恶意请求，在系统端判断是否符合规则，比如 id<0，布隆过滤器
    缓存击穿：查询加for update，永不过期

  7. Redis 持久化机制
    RDB：内存快照，比较快，存储的是内容，可以 BGSave，fork 一个子进程存储，对于 Slave 可以一开始用这个同步，之后再以 AOF 的形式同步，开启时会影响并发性能。进行 BGSave 时服务可能会暂停几秒（需要注意）
    AOF：文件追加，比较慢，比如每隔一秒将操作追加到log中，开启 AOF 会导致吞吐量降低，但是比 RDB 更完整的保护数据，可以将 AOF 文件转成 RDB。

  8. Redis 与 Memcached 的区别
    redis：支持多种数据结构，支持系列化，key-value 可存储上限为 1G,单核
    memcached：只支持 String，不支持序列化，value 大小最大为 1MB，支持多核

  9. Redis 并发竞争 key
    消息队列串行化
    zookeeper 的分布式锁实现：监听前面一个节点，当自己为第一个节点时，获得锁
    Redis 的分布式锁实现：setnx + 看门狗，比如当 key 过期快到的时候还没有删除，自动续时

  10. Redis 与 MySQL 双写一致性
    先写 MySQL，后写 Redis，如果担心 MySQL 写入速度慢（优化后可达 25000 条/秒），可以使用 Kafka 削峰

Zookeeper

  1. ZAB 协议
    消息广播：半数 follower 收到请求即 commit
    崩溃恢复：leader 崩溃后选举 leader的过程

  2. Leader 选举算法和流程
    每个ZooKeeper都有两个Id，一个是代表自己的Pid，一个是代表本身所存储的数据的Zid，
    一开始还没有leader也没有数据的时候会选举最大的Pid当leader
    当运行突然崩溃时，先每个都选举自己，广播自己的Pid和Zid，然后收到的机器广播最大的那个Zid对应的Pid，当半数+1 ZooKeeper 同意的时候即成为 Leader
