消费者组重平衡过程

消费者组是 Kafka 提供的可扩展且具有容错性的消费者机制，在 Kafka 中，消费者组是一个由多个消费者实例构成的组。多个实例共同订阅若干个主题，实现共同消费。同一个组下的每个实例都配置有相同的组 ID，被分配不同的订阅分区。当某个实例挂掉的时候，其他实例会自动地承担起它负责消费的分区。

当组成员发生变化、主题数量变化或者分区数变化，都会发生消费者组重平衡。在整个重平衡过程中，组内所有消费者实例都会暂停消费。

这个时候由 Broker 端的协调者组件来控制消费者组的状态流转，Kafka 为消费者组设计了一组状态机来帮助协调者完成整个重平衡流程。
### 消费者如何通过偏移量找到消息？
生产者发送到 Broker 的消息存储在 ${Kafka.home}/logs/主题_分区号 目录下的 .log 文件，如下。
每个 log 文件按段（Segement）分割成多个文件，每个段根据最小的消息偏移量（Offset）命名。也就是说，Kafka 每次顺序追加写 log 文件，当 log 文件的大小超过配置项指定的 log.segment.bytes 或者 log.roll.hours 时，自动创建一个新文件写入。

假设，topic-partition0 目录下有 segement1、segement2、…、segement n 个段。消费者向 Broker 发送一条消费 Offset 为 368801 的 message，Broker 是如何定位到消息位置的？

如上图，Broker 根据 368801 从 topic-partition0 目录通过二分法定位 .index 文件，因为 .index 文件是按最小的 Offset 顺序命名的，所以 368801 在 000000000000368796.index 里面。在这个 index 文件中，Kafka 使用稀疏索引（上图的 1,0、4,256、7,468 之间跳过了部分索引）来定位消息在 .log 文件的物理位置，368801 相对于 368796 的相对索引号为 5，通过二分查找获取到相对 Offset 为 4 的位置，从中获取实际的物理位置 256。最后，从 000000000000368796.log 直接找到物理位置 256 的消息，根据消息的 Offset 顺序往下查找，最后找到 368801 的消息。整个过程可以精简为 .index 二分法 -> .log 二分法 -> 顺序遍历。

如何确保消息不丢失？

Kafka 只能对已提交的消息做有限度的持久化保证，保证消息不丢的前提是 Kafka 的若干个 Broker 成功地接收到一条消息并写入到日志文件，这个过程中至少有一个 Broker 存活。通过调整 Kafka 的配置项，在非极端情况下，可以避免消息丢失的问题。

Spring Kafka 提供了相关的配置项，可以设置 Producer 的重发次数和 Consumer 手动提交 Offset。

生产者在发送消息时处理回调事件，如果接收到失败事件，根据业务场景决定是否重试。

消费者实例手动提交 Offset，为了避免消息积压，尽可能短事务执行。

如何保证消息顺序性？

首先需要明白，Kafka 是怎么存消息，无非就是将消息写入 logs 日志文件，正确的来说，是按顺序追加尾部的方式写到 Topic 下的一个分区，每写入一条消息会记录对应的 Offset，这个 Offset 是相对这个分区的自增序号，因此，单分区的情况下，消息是一定有序的，而多个分区就不一定能保证有序了。使用多分区是为了提高 TPS，有没有办法支持多分区（高性能）又能保证顺序消费（准确性）？Kafka 在发送消息时提供了一些路由机制，如下。

消息积压了怎么办？

消息积压的问题一般是消费者跟不上生产者的速度导致，一般我们通过 Java 程序来部署消费者实例，Java 应用完成一个增删改的事务往往需要用到数据库，例如 MySQL 数据库，双 1 配置下撑死每秒写入 1500 条就算不错了，也就是说 Java 应用的性能瓶颈一般是在数据库这边，假设生产者每秒发送 5000 条，消费者每秒只能消费 1500 条，按这样的消费速度，每秒将产生 3000 多条积压消息。解决消息积压的思路可以从数据库的优化来提高消费者的 TPS、使用多线程来消费 Topic。为了支持并发消费，需要修改产生积压的 Topic，将分区数调整为合适的数值 n，需要注意的是，消费者实例不能超过这个数值，否则会有一部分消费者阻塞无法使用。

在 Spring Kafka 从单次消费拉取一条消息改为几百条，并提高消费的线程数，可以大大提高消费的速度。
