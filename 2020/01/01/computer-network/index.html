网络基础

如何理解 OSI 和 TCP/IP 的区别？

OSI 模型：应用层 -> 表示层 -> 会话层 -> 传输层 -> 网络层 -> 数据链路层 -> 物理层
TCP/IP 协议：应用层 -> 传输层 -> 网际层 -> 网络接口，针对 OSI 作了精简

TCP/IP 各层常用的协议有哪些？

  1. 应用层：HTTP（网页）、FTP（文件）、SMTP（邮件）、WebSocket（持久双向通信）、RPC（Dubbo、gRPC）
  2. 传输层：TCP、UDP
  3. 网际层：IP、ARP（广播）、ICMP（ping 命令）
  4. 物理层：802

HTTP 有哪些版本特性？

  1. HTTP 1.0：已弃用
  2. HTTP 1.1：
    • 使用管道机制，在一个 TCP 连接允许同时多个请求，例如请求 A 和 B，但请求 A 阻塞会导致请求 B 阻塞（浏览器客户端在同一时间针对同一域名的请求有一定数据限制，超过限制数目的请求会被阻塞），即头队列阻塞
    • 支持 Range 断点续传
    • 支持 KeepAlive 持久连接
  3. HTTP 2.0：
    • 使用多路复用避免头队列阻塞
    • 使用二进制格式 “头信息帧+数据帧” 代替 “报文首部+报文主体”
    • 头部压缩，使用管道会对相同的头部 HPACK 计算，维护一张头信息索引表
    • 预读推送，例如读取一个 HTML，服务器会预读相关的 CSS、JS 文件推送给客户端
    • TLS 升级为 1.2
  4. HTTP 3.0：
    • 为了解决 TCP 失败重传导致同一个多个请求重传的问题，使用 UDP + QUIC 协议代替 TCP
    • 头部压缩改为 QPACK 算法
    • TLS 升级为 1.3

HTTP 有哪些常用的状态码：

  1. 2XX：请求成功
    • 200：请求成功，不代表业务处理成功
    • 201：请求成功，服务已成功创建，一般用于 POST 请求创建实体
    • 204：请求成功，但不返回内容、一般用于心跳维持
  2. 3XX：网页重定向
    • 302：临时重定向、例如通过短链接重定向到原始链接
  3. 4XX：客户端错误
    • 400：请求参数非法
    • 401：未认证
    • 403：未授权
    • 404：请求的路径不存在
  4. 5XX：服务端错误
    • 500：服务器错误
    • 502：网关错误
    • 503：服务端不可达
    • 504：网关超时

GET 和 POST 请求的区别

  1. GET 请求的查询参数可以被浏览器缓存，长度受浏览器限制，POST 请求的参数放在 Body，需要手动缓存，数据不受长度限制，但要考虑带宽和 TCP 拆包问题，另外 GET 请求的查询参数如果带中文，需要进行 URL 编码，而 POST 请求支持多种编码。
  2. 从 REST 的规范看，GET 请求只获取而不修改资源的信息或者状态，具备幂等性，而 POST 请求在 REST 服务表示新建一个资源，不具备幂等性。
  3. GET 请求只产生一个 TCP 连接，直接发送 HTTP Header 和查询参数，服务器直接响应期望的 200 状态码，而 POST 请求会产生两次 TCP 连接，首先发送 HTTP Header，服务器响应状态码为 100（continue），然后再发送 HTTP Body，服务器响应期望的 200 状态码。所以，遇到网络不好的情况下，POST 请求要比 GET 请求慢。
  4. GET 请求支持浏览器后退操作，而 POST 请求点击后退会提示重新提交表单。

如何理解三次握手的过程？

  1. 发送端 -> SEQ=0，SYN 同步位 -> 接收端，发送端此时进入 SYN_SEND 状态
  2. 接收端 -> ACK=SEQ+1=1，SYN 同步位，SEQ=0 -> 发送端，接收端此时进入 SYN_RECV 状态
  3. 发送端 -> ACK=接收端的 SEQ+1=1 -> 接收端，发送端和接收端都进入 ESTABLISHED 状态

两次握手无法保证网络发生时延的双向确认，四次握手没问题，但没必要。

如何理解四次挥手的过程？

  1. 发送端 -> SEQ=0，FIN 结束位 -> 接收端，发送端此时从 ESTABLISHED 状态进入 FIN_WAIT1 状态
  2. 接收端 -> ACK=SEQ+1=1 -> 发送端，接收端从 ESTABLISHED 状态进入 CLOSE_WAIT 状态，等待浏览器调用 close()，发送端此时从 FIN_WAIT1 状态进入 FIN_WAIT2 状态
  3. 接收端 -> FIN 结束位，SEQ=0 -> 发送端，接收端已经 close 了，进入 LAST_ACT 状态，等待发送端最后一次确认
  4. 发送端 -> ACK=接收端的 SEQ+1=1 -> 接收端，发送端进入 TIME_WAIT 状态，等待 2MSL 后进入 CLOSED 状态，接收端收到 ACK 后进入 CLOSED 状态。

发送端在 TIME_WAIT 状态等待 2MSL（2 个 报文最长的存活时间），是为了防止在四次挥手之前，接收端返回 SEQ 延迟，如果 TIME_WAIT 较短，在下一个三次握手时，这个延时的 SEQ 刚好到达，扰乱了握手的确认顺序。

TCP 如何保证传输可靠性？

  1. 确认和重传机制：建立连接时使用 序号、确认号 进行三次握手，传输过程中，如果 checksum 校验失败、丢包或者延时、发送端会重传
  2. 保证数据有序：在 TCP 报文中使用序号确保数据的有序性
  3. 流量控制：在 TCP 报文通过窗口大小确定双方收发的数据量大小
  4. 拥塞控制：使用 慢启动、拥塞避免、快速重传、快速恢复 算法避免占满带宽导致一部分数据包被丢弃。

如何解决 TCP 粘包/拆包？

首先要弄清楚为什么会有粘包和拆包的现象。TCP 默认有数据包的长度限制，发送数据报超过大小限制，被拆成两个包发送（拆包），然后进入发送端的缓冲区，通过层层转发，进入到接收端的缓冲区，而接收端读取缓冲区可能不及时，会导致将两次报文读在一块（粘包）。假如发送数据包 A=123、B=456，长度限制为 2，那么会进行拆包，接收端可能会接收到 A=12（被拆包了）、B=3456（读取不及时，粘包了）

  1. 收发端约定固定的长度发送或者读取，长度不超过限制，如果数据本身不固定长度，就会浪费空间，同时还会导致请求次数变多
  2. 使用分隔符来拆解，或者使用 JSON 的 {} 符号确定数据，缺点是需要读取所有的数据内容才能确定
  3. 在数据包前面增加一个标识报文长度的字段，收发端根据长度来截断报文，目前这个是最好的做法

TCP 如何实现流量控制？

TCP 通过滑动窗口控制收发端的流量，滑动窗口分为发送窗口和接收窗口，由 TCP 报文的 Window Size 设置：

  1. 发送窗口主要有 SND.UNA（已发送未收到 ACK 的位置）、SND.WND（发送窗口大小）、SND.NXT（可发送但未发送）
  2. 接收窗口主要有 RCV.NXT（待接收的位置）、RCV.WND（接收窗口大小）

滑动窗口还受操作系统缓冲区的占用而减少，从而影响流量的速率，可以通过下列参数配置：

  1. net.ipv4.tcp_adv_win_scale=1 // 应用缓存 = buffer / (2^tcp_adv_win_scale)
  2. net.ipv4.tcp_rmem=4096 // 读缓存，覆盖 net.core.rmem_maxs
  3. net.ipv4.tcp_wmem=4096 // 写缓存，覆盖 net.core.wmem_max
  4. net.ipv4.tcp_mem=1541646 // 系统无内存压力默认值，单位为页的数量
  5. net.ipv4.tcp_moderate_rcvbuf =1 // 开启自动调整缓存模式

TCP 如何实现拥塞控制？

拥塞控制以前根据丢包作为依据来实现：

  1. 慢启动：探测性检测是否丢失，默认发送 10 个 packet，每次收到一个 ack，拥塞窗口 cwnd 计数加一，接着按原来的 2 倍大小发送，呈指数增长。其中，发送窗口 swnd 从接收端的 window size 和拥塞窗口 cwnd 中取出最小值，公式为$swnd=SMSS*SMSS/cwnd$
  2. 拥塞避免：由于慢启动是指数发送的，发送次数越多，丢失的 packet 越多，为了避免这个情况，当慢启动达到阈值 ssthresh（slow start threshold）的时候，以线性的方式增加 cwnd，公式为 $cwnd=SMSS*SMSS/cwnd$，根据当前发送的大小减去一半，然后改为累加操作，呈线性增长，这个时候丢失的 packet 会很小
  3. 快速重传：拥塞避免减少了 packet 的丢失，这些丢失的 ACK 还可以通过快速重传来补充，例如，发送端发送了 pk0、pk1、pk2，其中 pk0 收到 ack1，pk1 丢失了，pk2 发到了接收端，接收端从 ack1 知道下一个值是 ack2，但是收到了 pk2，所以通过快速重传的规则，返回 ack2，接着发送端又继续发送了 pk3、pk4，接收端还是返回 ack2，此时发送到发送端收到了 3 次的 ack2，可以判断 pk1 是丢失了，这个时候立刻重发 pk1（如果超时重传到了时间点触发，就不会快速重传），接收端根据快速重传的规则，返回最后一个 pk4 期望的值 ack5。
  4. 快速恢复：由于到了快速重传这个阶段只是少量的丢包，此时的流量接近峰值，如果又重新回到慢启动就会降低网络的速率，因此，通过快速恢复来做适当的调整，首先将 ssthresh 设置为当前的 cwnd 降低为原来的一半，然后加上 3 个 MSS，在收到 ack5 之前每收到一个重复的 ack2，就增加 1 个 MSS，ack5 到达后就设置为 cwnd 为 ssthresh。

快速重传的算法有：

  1. 积极悲观：Go-Back-N，也称作 Reject（REJ）
  2. 保守乐观：selective repeat ARQ，也称作 Selective Reject（SREJ）
  3. SACK 算法：发送端在 TCP 报文的 options 设置 SACK，用 left edge - right edge 表示已收到的 ack，在 ack number 设置未收到的 ack（小于 left 和 right）

后来谷歌研发了 BBR 算法，以探测带宽作为依据，Linux 4.9 支持了这个功能。原理是通过多次测量获取 RTprop 和 BtlBW，如果链路发生了变化，通过 pacing_gain 拥塞来回调整计算出变化。

网络工具

网络抓包工具有哪些？

  1. WireShark：主要用于 Windows，分为捕获过滤器（选网络接口的时候输入）和显示过滤器（捕获后输入）
  2. tshark：WireShark 的 Linux 版本
  3. tcpdump：Linux 主要的抓包工具
  4. Chrome 的 F12 调试工具：主要用于 HTTP/WebSocket 协议

网络开发工具有哪些？

  1. Postman：用于 HTTP 开发
  2. Sockit：用于 Socket 开发
  3. SoapUI：用于 Webservice 开发
