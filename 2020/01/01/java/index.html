Java 基础

Java 有哪些特性？

  1. 抽象：使用数据实体表示现实生活的某一事物。
  2. 封装：将内部的数据和方法封装起来，对隐藏信息使用 private，公开接口使用 public。
  3. 继承：复用，抽取共同特性，类只能单继承，但接口可以支持多继承。
  4. 多态：方法的重载、类的继承 @Override，使不同类的对象对同一个消息做出不同的响应。

子类能否继承父类的构造方法？

子类不可以继承父类的构造方法，只可以调用父类的构造方法。子类中所有的构造函数都会默认访问父类中的空参数构造函数，这是因为子类的构造函数内第一行都有默认的 super() 语句。从内存实现或者反射的角度，子类继承所有父类的所有数据域和方法（拥有），但没有不具备对某些私有数据域或私有方法的使用权（调用）。

命名标识符可以包含哪些？

标识符可以包含英文字母、0 到 9 的数字，$ 以及 _，但不能以数字开头，同时标识符不能是 Java 的关键字。按照行业的命名规范，常量全大写，下划线隔开，类名用大驼峰式，变量名和方法名使用小驼峰式。

Java 基本类型有哪些？

基本类型 大小（字节） 默认值 包装类
byte 1 (byte)0 Byte
short 2 (short)0 Short
char 2 \u0000(null) Character
int 4 0 Integer
float 4 0.0f Float
long 8 0L Long
double 8 0.0d Double

基本类型在声明时系统会自动给它分配空间，而引用类型声明时只是分配了引用空间，必须通过实例化开辟数据空间之后才可以赋值。例如，数组对象也是引用类型，将数组复制到另外一个数组只是复制了一个引用，对另外一个数组的修改对原来的数组也是可见的。另外，Java 虚拟机没有提供 boolean 值专用的字节码指令，boolean 在编译之后都使用 int 基本类型来代替，占 32 位，而 boolean 数组将会被编码成 byte 数组，每个 boolean 元素占 8 位。

Java 运算符的分类和优先级

  1. 赋值运算符：=
  2. 比较运算符：<、>、==
  3. 算数运算符：除了 +、-、*、/、%，还有 i++ 和 ++i
  4. 逻辑运算符：&& 短路与、|| 短路或、! 非
  5. 移位运算符：用 << 表示二进制位往左移移动 n 位，后面补 0，实际上就是对 2 的取模少算 n 次，即乘以 $2^n$，同理，>> 表示除以 $2^n$。
  6. 位运算符：& 与、| 或、^ 异或
  7. 三元运算符：语法为 条件表达式? 表达式1：表达式2

从性能角度来看，在计算数值时，使用移位运算符、位预算直接操作二进制会更快，因为 CPU 的指令集是针对 1 和 0 操作的，加减乘除的运算需要更多的时钟周期。

对象引用的区别和场景

  1. 强引用：即使内存不足发生 OOM 时不会被回收，通过 new 方式或者反射机制完成分配。
  2. 软引用：每次 GC 时如果内存不足时会被回收，通过 SoftReference 创建，适用于缓存场景。
  3. 弱引用：每次 GC 时不管内存是否不足都被回收，通过 WeakReference 创建。
  4. 虚引用：在被回收之前，会被放入 ReferenceQueue 中，通过 PhantomReference 表示，一般用于 GC 统计，NIO 使用虚引用实现了堆外内存的监视。

分析 String 类的源码

  1. String 使用 final 修饰，不可继承。
  2. String 内部维护了一个 char[] value 数组，但是可以通过反射机制进行破坏，非绝对安全。
  3. String 使用 == 比较的是对象地址，比较内容使用 equals()，时间复杂度为字符串的长度，对于超大字符串比较是否相等，使用对象地址 intern() 来比较会好一些。
  4. String 的 equals() 方法逐个比较字符的内容是否相等，hashCode() 方法逐个字符 * 31 并累加获取哈希值，存在哈希碰撞的可能性。
  5. String 的 intern() 方法尝试从堆中的字符串对象池获取，如果没有，就从堆获取，还是没有，就新建对象，放入字符串对象池，并返回这个地址。
  6. String 的 compareTo() 方法从左到右比较每个字符的大小，如果都相等，就比较字符的长度

理解 equals 和 hashCode 方法

  1. Object 类的 equals() 比较的是对象地址，hashCode() 使用 native 比较哈希值。
  2. String 类在没有覆盖方法的情况下，equals() 相等，hashCode() 一定相等，hashCode() 相等不能保证 equals() 相等，因为 hashCode() 是根据 equals() 的内容来生成哈希，意味着哈希碰撞。

Java 特性

Java 8 有哪些特性？

  1. Optional：解决空指针
  2. FunctionalInterface：函数式接口
  3. 接口新增了 default 修饰符，允许在接口实现默认方法
  4. Lambda 表达式：例如针对函数式接口使用 () -> {} 来简化代码
  5. 方法引用：使用 :: 引用类或者对象的方法
  6. 抽象了 Stream 的 API操作：stream() 串行和 parallelStream() 并行，使用 parallelStream() 需要保证线程安全。

Java 集合

Java 常用的集合类有哪些？场景？

从数据结构分为线性结构和散列结构，Collection 接口是线性结构，子类包含 List 列表、Queue 队列、Set 集合，而 Map 接口属于散列结构。这些数据结构的底层无非就是使用数组或者链表实现存储，使用数组的优势在于查找某个元素的时间复杂度为 $O(1)$，但不适合插入、删除、扩容，使用链表可以使某个元素的插入和删除的时间复杂度为 $O(1)$，但不适合查找某个元素，最多可以优化为通过头尾指针实现双端操作。

  1. 列表：元素可以重复，有序，使用数组存储

    • ArrayList -> List -> Collection：使用数组存储，初始大小默认为 10，每次扩容增大 1.5 倍，非线程安全
    • Vector -> List -> Collection：同样使用数组存储，和 ArrayList 的区别就是每个方法都是用 synchronized 加锁来保证线程安全，并且每次扩容增大 2 倍
    • CopyOnWriteList -> List -> Collection：相对 Vector` 的性能优化，读取不加锁，写时加锁再复制数组，不适合大数据量的存储
    • Stack -> Vector：遵循 LIFO 原则实现了栈的操作，如 push()、pop()、peek() 方法，缺点是每个方法都用 synchronized 加锁来保证线程安全
    • LinkedList -> List、Deque（Queue）：相对 Stack 的性能优化，支持双端操作

  2. 集合：元素不重复，无序/有序

    • HashSet -> Set -> Collection：使用 HashMap 的 key 存储元素，value 存一个内部的 Object，获取一个元素的复杂度为 $O(1)$。
    • LinkedHashSet -> Set -> Collection：继承了 HashSet，使用 LinkedHashMap 的 key 存储元素
    • TreeSet -> Set -> Collection：基于 NavigableMap 和 Comparator 实现有序性

  3. 队列：遵循 FIFO 先进先出原则

    • ArrayBlockingQueue -> Queue -> Collection：有界队列，使用对象数组存储，适用于消息队列 LRU 算法
    • LinkedBlockingQueue -> Queue -> Collection：无界队列，使用链表存储
    • PriorityBlockingQueue -> Queue -> Collection：优先队列，使用对象数组存储，通过 Comparator 排列优先级，注意 PriorityQueue 这个类不是线程安全的
    • DelayQueue -> Queue -> Collection：延迟队列，使用对象数组存储，通过 Delay 接口实现延时，适用于订单支付倒计时

  4. 散列：查找单个元素快，但需要注意哈希碰撞的问题

    • HashMap -> Map：使用数组存储 Map.Entry 元素，下标为 key 的哈希值，哈希碰撞时转换为链表，碰撞次数达到 8 时转化为红黑树，缩容为 6 时退化为链表，非线程安全
    • LinkedHashMap -> Map：继承 HashMap，额外维护双向链表，多了 head 头引用、tail 尾引用、before 前一个引用、after 后一个引用
    • TreeMap -> Map：借助 NavigableMap 组合 Comparator 存储
    • Hashtable -> Dictionary、Map：和 HashMap 是实现类似，但是每个方法都用 synchronized 加锁来保证线程安全

深入理解 HashMap 的数据结构

  1. 使用数组存储 Map.Entry 元素，下标为 key 的哈希值
  2. 哈希值计算先右移 16 位，再进行与运算
  3. 哈希碰撞时转换为链表，碰撞次数达到 8 时转化为红黑树，缩容为 6 时退化为链表。
  4. 元素个数的初始大小为 16，里面还有个负载因子为 0.75，这是一个经验值，在超过 75% 时，哈希碰撞概率大，性能急剧下降，需要扩容
  5. 非线程安全，并发场景下 rehash 操作会发生闭环，使用 ConcurrentHashMap 来保证线程安全

ConcurrentHashMap 是怎么保证线程安全的？

ConcurrentHashMap 在 JDK 1.7 时使用 Segement 分段锁，1.8 后改用 CAS 实现。

Java 多线程

进程和线程、协程有哪些区别？

  1. 进程：资源分配的基本单位，进程的上下文切换需要复制分配的堆、栈
  2. 线程：CPU 调度的基本单位，线程共享进程的堆，上下文切换只需要复制栈
  3. 协程：在用户态划分更小的执行任务

进程间怎么通信？

  1. 管道：匿名管道 | fork 子进程，命名管道 mkfifo，会阻塞窗口
  2. 消息队列：解决管道阻塞，异步消息发送，有大小限制，需要用户态和内核态的上下文切换，相关的函数有 msgget()、msgsend()、msgrcv()、msgctl()。
  3. 共享内存：减少用户态和内核态的上下文切换，相关的函数有 shmctl()、shmget()。
  4. 信号量：PV 原语操作，解决共享内存的同步问题，假设信号量 S 为 0，当信号量大于等于 0 可以进入同步区，首先进程 A 访问同步区，因为 S = 0，执行 P 操作，S = S - 1 = -1，这个时候进程 B 访问同步区，由于 S < 0，无法进入，只能阻塞等待，当进程 A 从同步区出来，执行 V 操作，S = S + 1 = 0，由于 S = 0，原来阻塞的进程 B 就可以访问同步区了，接着执行 P 操作…
  5. 信号：kill -l 发送终止信号或者 CTRL+C 中断，进程间唯一的异步通信机制
  6. Socket：分为远程 Socket 和本地 Socket，本地 Socket 会锁定一个 Socket 文件

如何保证线程安全？原理？

处理线程安全的方法有三类：避免资源竞争、内存可见性、锁机制。

  1. 避免资源竞争
    • 每次调用都 new() 实例化一次，性能最差，不可取
    • 使用局部变量，每个线程进入一个方法会独立保存自己的栈变量，没有冲突问题
    • 全局变量只读操作，分时访问（白天读，半夜写），分段访问（修改变量集合的某一段，不修改总长度）
    • 全局变量保存在 ThreadLocal 类，底层把变量保存到当前线程的内部变量 ThreadLocalMap，实际上就不是共享资源了，而是每个线程访问自己的资源。例如 Hibernate 使用 ThreadLocal 来保存 Session。
  2. 内存可见性
    • Java 提供了volatile 关键字来修饰共享变量，编译成字节码后增加 Lock 前缀指令，强行让 CPU 从主存读取数据，保证线程 A 在线程 B 读取之前修改，CPU 将修改操作刷新到主存，线程 B 读取时 CPU 不再从高速缓存读取，而是从主存读取，再刷新到缓存，注意 volatile 修饰的共享变量不能保证原子性操作，例如 i++ 语句对于底层指令来说，先 i+1，再赋值给 i，这是两个步骤，非线程安全
    • 为了解决 volatile 的不足，使用并发包的原子类实现自增操作，例如 AtomicInteger，使用内部的 incrementAndGet() 方法，底层实际上是使用 Unsafe 的 CAS 操作
  3. 锁机制
    • 使用 synchronized 关键字，支持类、对象、代码块的同步操作，编译成字节码后增加 MonitorEnter 和 MonitorExit 监视器，首先，如果线程 A 进去这个同步块，如果没有其他线程竞争，会在代码记上 Mark 偏斜锁，下一次进来直接判断这个标记就可以进入了，提高性能，如果后续有线程 B 竞争，偏斜锁被撤掉，线程 B 通过自旋获取锁，升级为轻量级锁，如果线程 B 自旋一定的次数还拿不到锁，此时所有的线程都被阻塞，升级为重量锁。
    • 由于 synchronized 没有阻塞超时机制，如果没有释放锁，其他线程将永远得不到锁，甚至使用不当，还会发生死锁，这个时候要祭出并发包 ReentrantLock 可重入锁，不仅提供了超时机制，还提供了公平锁和非公平锁的操作，底层利用 Unsafe 的 CAS 操作来控制 state，如果 CAS 失败，就加入 AQS 队列，调用 Unsafe 的 park 挂起线程，加入 wait 队列，等待自旋 CAS 操作成功后，再调用 Unsafe 的 unpark 唤醒，另外，synchronized 使用 Object 的 wait() 和 notify() 来实现等待和唤醒，ReentrantLock 使用 Condition 的 await() 和 signal() 实现类似的功能。
    • 对于读多写少的场景，可以使用 ReentrantReadWriteLock 实现读读不互斥，读写和写写互斥。
    • JDK 1.8 还提供了支持乐观锁 StampedLock，底层并没有继承 AQS，而是根据自己的规则进行 CAS 操作。
    • 无论是 ReentrantLock 还是 synchronized，只能锁定同一个资源，如果需要允许一定的线程数访问资源，那么可以使用 Semaphore 信号量，在初始化的时候指定最大访问容量，底层使用 AQS 根据容量来控制访问数，在访问资源的前后，使用 acquire() 申请锁，然后使用 release() 释放锁
    • 如果一个线程需要等待多个线程执行完成才会继续执行，那么可以使用 CountDownLatch 计数器，通过构造函数初始化计数，主线程使用 await() 等待其他线程执行 countDown() 减 1，当计数为 0 时，主线程继续执行。
    • 如果多个线程需要共同等待某个时机才往下执行，那么可以使用 CyclicBarrier，通过构造函数初始化这个线程的数量（管理到同一个组），每个线程在对应的时机执行 await()，那么，线程都会阻塞，直到所有线程执行到这个方法，所有线程同时往下执行。
    • 并发包还提供了一些容器的线程安全类，例如 CopyOnWriteArrayList 写时复制，读不加锁，写时加锁并调用 System.copyArray() 复制到新的数组，适用于数组不是很大的场景，还有 ConcurrentHashMap、ConcurrentHashSet 通过分段锁 + CAS 实现散列和集合的安全访问，另外，涉及到队列的有 ArrayBlockingQueue 有界队列、LinkedBlockingQueue 无界队列、PriorityBlockingQueue 优先队列、DelayQueue 延时队列，底层都是 ReentrantLock 实现同步。

Thread 和 Object 有哪些线程的方法？区别？

首先，需要知道 Unix 的线程调度是时间片算法，Windows 是抢占式算法。

  1. Thread.sleep(TimeUnit)：指定时间 n 暂停执行，让出当前 CPU 时间片，不释放锁，暂停期间会抛出中断异常，且能被中断，时间 n 到，重新参与 CPU 竞争，sleep(0) 也会触发 CPU 竞争
  2. Thread.yield()：暂停执行，让出当前 CPU 时间片，不释放锁，暂停期间不会抛出中断异常，重新参与 CPU 竞争
  3. Thread.join()：线程 A 调用线程 B 的 join()，线程 A 要等线程 B 执行完才能继续执行
  4. Object.wait()：需要通过 synchronized 锁住某个对象，这个对象调用 wait() 就会让出 CPU 时间片和释放锁，只能等待其它线程唤醒
  5. Object.notify()/Object.notifyAll()：同样需要通过 synchronized 锁住某个对象，notify() 随机唤醒一个线程，notifyAll() 唤醒所有线程，线程被唤醒后回到就绪状态，等待 CPU 调度

线程状态有哪些？

新建 -> 就绪 -> 运行 -> 阻塞/结束

  1. 新建：new Thread()
  2. 就绪：start()、Object.notify() 唤醒、Thread.yield() 重新竞争、Thread.sleep() 睡眠时间到，重新竞争
  3. 运行：得到 CPU 调度的时间片。
  4. 阻塞：等待 I/O、等待锁、Object.wait() 让出 CPU 和释放锁、等待其它 join 线程执行完
  5. 结束：时间片用完、异常结束、run() 方法执行完毕。

怎么合理使用线程池？

Java 提供了 Executors 工具类来创建线程池，但是不适用于高并发场景：

  1. newFixedThreadPool（固定线程数）：当线程数超过固定线程数时，会加入无界队列 LinkedBlockingQueue，高并发下会 OOM
  2. newCachedThreadPool（0 ~ Integer.MAX）：如果单个线程执行的时间很长，就会没法及时复用已有的线程，高并发下创建大量的线程，每分配一条线程占用 1MB，会发生两种情况，一个是 OOM，一个是内存在一段时间内占用很大，后面又降下来了，这是因为 newCachedThreadPool 默认设置了 60s 的超时，60s 后会释放一些线程
  3. newSingleThreadExecutor（固定 1 个线程）：
  4. newScheduledThreadPool（延迟启动）：DelayedWorkQueue（自旋锁延迟）

线程池应该根据场景来指定线程核心数、线程最大数、线程等待队列大小（缓冲），使用 ArrayBlockingQueue 有界队列来避免 OOM

  1. 单核 CPU 不要用多线程，避免不必要的上下文切换
  2. CPU 密集型：线程数不要太大，可以是 CPU 核心数 + 1，多一个线程是为了最大利用某个线程暂停时浪费的时间，同时，为了避免 CPU 线程数不沟通而产生阻塞，应当使用较长的队列做缓冲
  3. I/O 密集型：线程数为 CPU 核心数的 2 倍
  4. 多核普通任务：线程数除了 CPU 核心数，还要加上线程等待 I/O 时间占总运行时间的百分比，即 $CPU 核心数 * (1 + 线程等待 I/O 时间/线程总运行时间)$

Java 并发

常用的并发类有哪些？原理和场景？

  1. 并发容器
    • ConcurrentHashMap：JDK 1.7 使用 Segement 数组分段锁，1.8 后使用 CAS 实现。
    • CopyOnWriteArrayList：读不加锁，写操作调用 System.copyArray 拷贝数组，适用于读多写少，数据量本身不大的场景
  2. Atomic
    • AtomicInteger/AtomicLong/：提供原子性操作的方法，使用 Unsafe 的 CAS 实现
    • AtomicReference：适用于对象，同样使用 Unsafe 的 CAS 实现
  3. Lock
    • AbstractQueuedSynchronizer：AQS，阻塞队列，Lock 包实现的核心类
    • ReentrantLock：可重入锁，内部使用 Sync 继承 AQS，当获取不到锁就调用 Unsafe 的 park 挂起线程，加入 wait 队列，等待自旋 CAS 操作成功后，再调用 Unsafe 的 unpark 唤醒，和 synchronized 对比，增加了公平锁和非公平锁的支持，支持等待锁的超时
    • ReentrantReadWriteLock：可重入读写锁，读读不加锁，读写和写写互斥

了解过哪些锁？

  1. 乐观锁：无锁，CAS 比较并交换，例如 AtomicInteger、AtomicReference
  2. 悲观锁：synchronized 关键字、Lock 接口
  3. 读锁（共享锁）：ReentrantReadWriteLock
  4. 写锁（排它锁）：ReentrantLock、synchronized 关键字
  5. 可重入锁：ReentrantLock、synchronized 关键字
  6. 不可重入锁：同一个线程进入同一个方法不能再次获取锁
  7. 公平锁：ReentrantLock.FairSync -> AQS，排队
  8. 非公平锁：ReentrantLock.NonfairSync -> AQS，在初始化的时候尝试先插队，失败就排队
  9. 偏斜锁 -> 轻量级锁 -> 重量级锁：特指 synchronized 的锁升级，线程 A 进入时，在竞争不激烈的情况下得到偏斜锁（Mark word，缓存），如果后续线程 B 竞争，JVM 撤销偏斜锁，升级为轻量级锁，此时线程 B 自旋获取锁（占用 CPU，避免线程上下文切换），自旋到一定的次数如果还获取不到锁，就退出自旋（让出 CPU），此时升级为重量级锁，所有等待的线程都阻塞。

以下是 MySQL 的锁类型，和 Java 的读写锁是共性的

  1. 元数据锁（MDL）：修改表结构会锁住
  2. 表锁：MyISAM 引擎默认的锁粒度，还有 InnoDB 未使用索引也可能产生表锁
  3. 行级锁：InnoDB 引擎支持
  4. 间隙锁：使用 where a = 1 锁的是记录为 1 的数据行，使用 where a between 1 and 2 锁的是区间
  5. Next Key Lock：间隙锁 + 行级锁，将间隙锁的开区间改为闭区间
  6. 乐观锁：在表结构维护一个版本字段
  7. 悲观锁：显示加锁，读锁 + 写锁
  8. S 锁：select * from t lock in share mode，同时读不冲突，读写会冲突
  9. X 锁：select * from t for update，无论是读还是写都会冲突
  10. 意向锁：分为 IS 锁和 IX 锁，由 MySQL 自动添加，和行级的 S 锁和 X 锁不冲突，只会和表级的 S 锁和 X 锁发生冲突。

怎么触发死锁？如何检测和解决？

假设有 o1 和 o2 共享对象，线程 A 执行 synchronized(o1) -> synchronized(o2)，线程 B 执行 synchronized(o2) -> synchronized(o1)，两个线程互相等待对方的资源，不释放锁。

如何正确使用超时参数？

首先要弄清楚连接超时和读取超时的区别：

  1. 连接超时指建立连接所需要的时间，一般的配置项为 ConnectionTimeout，建议设置为 1~5 秒，HTTP 是基于 TCP 传输数据的，TCP 三次握手的时间很短，连接超时一般是网络或者防火墙配置的问题，如果在局域网，可以设置得更短。当发生超时，客户端会抛出 java.net.SocketException: connetct time out 的异常。
  2. 读取超时指的是建立连接后从服务器读取到可用资源所用的时间，一般的配置项为 ReadTimeout ，建议设置不要超过 10 秒，以免高并发场景产生大量的阻塞请求。当发生超时，客户端会抛出 java.net.SocketException: read time out 的异常，服务器有可能会继续执行，也有可能会抛出 Client Abort 的异常。
  3. 注意下框架默认的超时配置，例如 Ribbon，默认配置 1 秒的连接超时和 1 秒的读取超时，前者还好一些，后者成功率很低，一般读取的耗时跟数据库有关，遇到事务处理，估计也要 3~5 秒，所以，使用 Ribbon 时应当配置：

Java 虚拟机

如何理解 Java 内存模型?

  1. 程序计数器：线程独占，不会发生 OOM，任意时刻线程只会执行一个方法的代码，如果不是 native 的，就存放当前正在执行的字节码指令的地址，如果是 native，则为 undefined。
  2. 虚拟机栈：线程独占，与线程同时创建，每个方法在执行时在其中创建一个栈帧，用于存储局部变量、操作数栈、动态链接、方法返回地址。正常调用完成后恢复调用者的局部变量表、操作数栈、递增程序计数器来跳过刚才执行的指令，或抛出异常不将返回值返回给调用者，JDK 1.5 开始每个线程占 1M，可以通过 -Xss 修改，当调用栈超过上限值，会抛出 StackOverFlow 异常
  3. 本地方法栈：线程独占，由 C/C++ 语言实现的，在 Java 代码使用 native 标记，从用户态向内核态发起系统调用，借助 C 语言实现，例如线程的操作、Unsafe 执行 CAS 或者额外内存申请。
  4. GC 堆：线程共享，主要保存对象的实例，分为新生代和年老代，新生代又细分为 eden、survior1、survior2，比例为 8：1:1，因为新生代使用复制回收算法，所以实际上使用的是 9 成，剩下 1 成用于复制交换。可以通过 -Xms 或者 -Xmx 修改，超过上限会抛出 java.lang.OutOfMemoryError: Java heap space 异常，另外，JDK 1.7 将字符串常量池移到堆，在堆中是独立的一个空间
  5. 方法区：HotSpot 在方法区的规范实现了永久代，可以通过 -XX:PermSize 或者 -XX:MaxPermSize 修改，超过上限会抛出 java.lang.OutOfMemoryError: PermGen space 异常，在 JDK 1.8 后被移除，替换为元空间，受堆外的物理内存影响
  6. 元空间：替换永久代，通过 -XX:MetaspaceSize 或者 -XX:MaxMetaspaceSize 修改，超过上限会抛出 java.lang.OutOfMemoryError: Metaspace 异常

类加载过程是怎么样的？

  1. 加载：通过类的完全限定名，查找此类字节码文件，利用字节码文件创建 Class 对象，存储到方法区；
  2. 验证：验证字节流信息是否符合 JVM 规范，防止 Class 文件被篡改
  3. 准备：进行内存分配，为 static 修饰的类变量分配内存，并设置初始值（0 或 null），不包含 final修饰的静态变量，因为 final 变量在编译时分配
  4. 解析：将常量池的符号引用改为对指针地址的直接引用
  5. 初始化：执行类构造器、对类变量赋值、静态语句块。先初始化父类，再初始化当前类，只有对类主动使用时才会初始化
  6. 使用：实例化对象，分配内存空间。
  7. 卸载：GC 回收。

类加载的时机如下：

  1. Hot JVM 启动加载
  2. Class.forName：将类的 .class 文件加载到 JVM 中，并执行类中的 static 块
  3. ClassLoader.loadClass：只加载 .class 文件，不执行 static 块
  4. 反射：通过 Reflect 包的 API 调用
  5. JNI 调用类对象

类初始化条件有哪些？

  1. 使用 new 关键字实例化对象：遇到 new、getstatic、putstatic、invokestatic 这四条字节码指令时，如果类没有进行过初始化，则需要先触发其初始化。如的时候、读取或设置一个类的静态字段的时候，以及调用一个类的静态方法的时候。除了被final修饰已在编译器把结果放入常量池的静态字段，对于静态字段，只有直接定义这个字段的类才会被初始化，
  2. 反射：使用 java.lang.reflect 包的方法对类进行反射调用的时候，如果类没有进行过初始化，则需要先触发其初始化。
  3. 当初始化一个类的时候，如果发现其父类还没有进行过初始化，则需要先触发其父类的初始化；
  4. 当虚拟机启动时，用户需要指定一个要执行的主类 main() 方法，虚拟机会先初始化这个主类；
  5. 当使用 JDK1.7 动态语言支持时，如果一个 java.lang.invoke.MethodHandler 实例最后的解析结果 REF_getstatic、REF_putstatic、REF_invokeStatic 的方法句柄，并且这个方法句柄所对应的类没有进行初始化，则需要先出触发其初始化。

子类引用父类中定义的静态字段，只会触发父类的初始化而不会触发子类的初始化。
