MySQL 基础

建库建表需要注意哪些问题？

  1. 数据库字符集：高版本优先选择 utf8mb4，支持 emoji 表情
  2. 大小写敏感问题：Windows 默认不区分大小写、Linux 默认区分表的大小写，建议规范全部小写化。
  3. 表名和列名的长度都不能超过 64 位，如果为了兼容多种数据库和后期数据迁移，最好不要超过 30 位（Oracle 限制）。
  4. 建表不要使用 MySQL 的关键字和保留字，如 password、user、event、type 等等，否则抛出 You have an error in your SQL syntax 错误。

MySQL 的索引有哪些？用法？

  1. 聚集索引/主键索引：依次从 primary key -> 唯一索引 -> 隐藏的 6 字节 row_id 建立索引，索引用到 B+ 树，顺序写可以提高 I/O 性能，建议使用 MySQL 自增或者雪花算法。
  2. 非聚集索引/二级索引：查询时从二级索引树匹配 key 找到对应的主键 ID，然后从主键索引将 ID 作为 key 查找叶子节点，返回数据。
  3. 覆盖索引：如果二级索引本身包含了所有查询的字段，就不会再从主键索引查找（回表），建议业务查询的字段不多时，可以建立组合多个字段建立索引。
  4. 组合索引/联合索引：遵循左匹配原则，如果查询字段没有覆盖到创建索引的最左边字段，可能用不到索引。
  5. 唯一索引：对插入非 null 的值进行唯一性校验，重复插入会失败，建议使用唯一索引要设置为 not null。
  6. 簇集索引：指索引和数据都在同一份文件，InnoDB 的主键索引和数据行都在同一棵树，而 MyISAM 的索引和数据都是分开的，不属于簇集索引

怎么理解 MySQL 的事务？

MySQL 的 Server 层并不关心事务，由存储引擎 InnoDB 管理事务，事务遵循 ACID 特性：

  1. Atomicity（原子性）：事务要么执行成功，要么全部回滚
  2. Consistency（一致性）：事务前后保证数据完整性，例如转账操作，支出和收入是相等的。
  3. Isolation（隔离性）：多个事务的执行期间不会互相干扰
  4. Durability（持久性）：事务提交后保证持久化到磁盘

其中，MySQL 的事务隔离性又细分为：

  1. 未提交读（Read Uncommitted, UC）：最低隔离级别，还不如用 MyISAM
  2. 已提交读（Read Committed, RC）：解决脏读，Oracle 默认的隔离级别
  3. 可重复读（Repeatable Read, RR）：解决不可重复读，MySQL 默认的隔离级别
  4. 序列化（Serializable）：解决幻读，性能极差，不过 MySQL 在 RR 模式下通过间隙锁 gap lock (next-key lock) 解决了幻读的问题。

InnoDB 有哪些新特性？

  1. 双写（double write）：保证数据可靠性，由于 InnoDB 按页 16KB 存储，而 OS 一般是 4KB 写入操作的，可能会发生数据页写了 4KB 时发生断电，导致数据不完整，所以除了写入磁盘的数据页还要写入一份到共享空间，断电重启后检查校验值是否一致，否则从 redo-log 恢复数据。
  2. 插入缓冲（insert buffer）：提高写入性能，先插入 Buffer Pool，当写入的大小超过阈值或者刷盘时间到，就写入磁盘
  3. 自适应哈希索引（ahi）：提高查询性能，使用 = 等值查询时，MySQL 可能自动优化为 hash 去查

如何理解 MySQL 的锁机制？

  1. 元数据锁（MDL）：修改表结构会锁住
  2. 表锁：MyISAM 引擎默认的锁粒度，还有 InnoDB 未使用索引也可能产生表锁
  3. 行级锁：InnoDB 引擎支持
  4. 间隙锁（Gap Lock）：使用 where a = 1 锁的是记录为 1 的数据行，使用 where a between 1 and 2 锁的是区间
  5. Next Key Lock：间隙锁 + 行级锁，将间隙锁的开区间改为闭区间
  6. 乐观锁：在表结构维护一个版本字段
  7. 悲观锁：显示加锁，读锁 + 写锁
  8. S 锁：select * from t lock in share mode，同时读不冲突，读写会冲突
  9. X 锁：select * from t for update，无论是读还是写都会冲突
  10. 意向锁：分为 IS 锁和 IX 锁，由 MySQL 自动添加，和行级的 S 锁和 X 锁不冲突，只会和表级的 S 锁和 X 锁发生冲突。

MySQL 原理

MySQL 是怎么解决幻读的？

MySQL 使用快照读和当前读解决幻读。

  1. 快照读：MySQL 使用 MVCC 机制来保证数据的一致性，读取数据时不需要对数据进行加锁，且不会被其他事务阻塞。MVCC 在数据行使用 2 个隐藏的字段，事务 DB_TRX_ID 和 回滚指针 DB_ROLL_PTR。
    • 使用 RC 隔离级别时，在事务的每个 select 触发一个新的 DB_TRX_ID
    • 使用 RR 隔离级别时，在事务的第一个 select 触发一个新 DB_TRX_ID
  2. 当前读：也称为锁定读，需要在 SQL 语句上加锁，分为排它锁 select * from t for update 和共享锁 select * from t in share mode。

普通的 select 使用快照读，RR 隔离级别下 update 语句使用当前读，而 RC 隔离级别下 update 语句使用半一致性读（semi consistent），半一致性读指一个 update 语句，如果读到一行已经加锁的记录，返回最新记录，如果没加锁，就先发起一次查询，然后再加锁。

一条 SQL 查询语句是怎么执行的？

以 select name, age from T where ID = 10; 为例：

  1. 连接器收到 SQL 语句，先从查询缓存查找是否执行过，如果有，就直接返回（新版本已废弃查询缓存，因为每次更新操作会清空缓存，影响性能）
  2. 分析器对 SQL 进行词法分析，解析出哪些是字段、表
  3. 分析器对 SQL 进行语法分析，验证是否符合 MySQL 的规则
  4. 优化器判断 SQL 是否使用索引，选择哪个索引
  5. 执行器根据优化器的分析结果，先判断是否有执行权限，然后根据表的定义找到对应的存储引擎，调用这个存储引擎的接口，并返回结果

一条 SQL 更新语句是怎么执行的？

和 SQL 查询类似，分析器根据词法和语法分析知道这个是更新语句，然后通过优化器选取索引，查找对应的记录去更新，以 update T set age = age + 1 where ID = 2; 为例：

  1. 先取 ID = 2 这一行，查找 Buffer Pool 是否存在这行记录，如果有，直接返回，如果没有，从磁盘中查找并读取 Buffer Pool
  2. 执行器拿到存储引擎返回的数据，把这行数据更新，再调用存储引擎接口更新到 Buffer Pool
  3. 存储引擎将新数据更新到 Buffer Pool，同时将更新操作写入 redo-log，进入 prepare 阶段，通知执行器执行完成了，可以提交事务
  4. 执行器生成这个更新操作的 bin-log，并写入磁盘
  5. 执行器调用存储引擎的提交事务接口，把 redo-log 改为 commit 状态。

MySQL 实战

自增主键用完了怎么办？

自增主键分两种情况

  1. 指定自增主键：自增值达到上限时，申请的下一个 ID 还是原来的，这个时候会抛出重复主键的错误，一般使用 int 作为主键，上限值为 $2^{32}-1$（42 亿+），在设计库表时，应该评估下规模，如果很大，可以选择 bigint unsigned 类型。
  2. 未指定主键，使用隐藏的 6 个字节 row_id：上限值为 $2^{48}-1$，申请的下一个 ID 会归 0 再重新递增，如果出现相同的 row_id，后写的数据会覆盖之前的数据。

如果一开始时使用了 int 类型，可以修改为 bigint unsigned 类型，不过，发生这种情况，一定是系统的设计问题，单表超过 2000W 时，树的高度会超过 3，性能急剧下降，这个时候应当对数据进行分表/分库。
