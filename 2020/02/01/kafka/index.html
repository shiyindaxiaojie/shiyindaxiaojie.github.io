Kafka 基础

Kakfa 到底是什么？

Apache Kafka 是消息引擎系统，也是一个分布式流处理平台。业界还有企业级 Confluent Kafka、云平台 Cloudera/Hortonworks Kafka。消息引擎系统是一组规范。企业利用这组规范在不同系统之间传递语义准确的消息，实现松耦合的异步式数据传递。Kakfa 支持点对点模型和发布/订阅模型，同时，Kafka 并未完全遵循 JMS 的规范。

Kafka 有哪些概念？

  1. Producer（生产者）：发送消息到 Broker，可以轮询/哈希/指定分区发送，支持消息重发
  2. Consumer（消费者）：从 Broker 拉取消息，可以手动/自动提交 Consumer 偏移量
  3. Consumer Group（消费者组）：多个消费组构成同一个组，共同消费一个主题，随机消费/轮询消费/按分区消费
  4. Broker（消息代理实例）：在集群下等同于 Kafka 服务器节点，保证高可用
  5. Topic（主题）：消息所属的主题，文件形式表现为 .log，同目录下还有 .index
  6. Partition（分区）：将主题划分为 n 个分区，让 m（m<= n）个消费者并发消费，提高吞吐量
  7. Segement（段）：Kafka 顺序写入 Logs/Partitions，避免单个消息文件过大，按偏移量分割成多个段，再通过索引文件定位消息位置
  8. Offset（偏移量）：一个是指消息在分区的偏移量，另外一个是指消费者已消费的偏移量，高版本可以在 _consumer_offset 的主题查看
  9. LEO（最后偏移量）：每个副本都有 LEO，指的是最新的消息位置，Leader 的 LEO 对客户端可见，Follower 的 LEO 由于同步延迟，对客户端暂时可能看不到
  10. HW（高水位）：指的是所有的 Follower 同步 Leader 最慢的 LEO，HW 就是对客户端可见的临界点
  11. LSO（事务偏移量）：
  12. Replication（副本因子）：指的是 replication.factor 配置主题总共有多少个副本
  13. ISR（同步副本集合）：指的是副本因子在同步集合的个数，Replication >= ISR
  14. Leader/Follower：由 Zookeeper 协调器通过选举的共识算法指定分区哪个是 Leader 负责读写，哪个是 Follower（负责同步 Leader）

怎么理解 Kafka 消费者组？

消费者组是 Kafka 提供的可扩展且具有容错性的消费者机制，在 Kafka 中，消费者组是一个由多个消费者实例构成的组。多个实例共同订阅若干个主题，实现共同消费。同一个组下的每个实例都配置有相同的组 ID，被分配不同的订阅分区。当某个实例挂掉的时候，其他实例会自动地承担起它负责消费的分区。

Kafka 为什么要用 ZooKeeper？

目前，Kafka 使用 ZooKeeper 存放集群元数据、成员管理、Controller 选举，以及其他一些管理类任务。

  1. 存放集群元数据：主题分区集群相关的所有数据都保存在 ZooKeeper 中。
  2. Broker 管理：Broker 节点的注册、注销以及属性变更等等。
  3. Controller 选举：选举集群 Controller，而其他管理类任务包括但不限于主题删除、参数配置等。

后面的版本规划，Kafka 将完全不再依赖于 ZooKeeper，使用社区自研的基于 Raft 的共识算法，替代 ZooKeeper，实现 Controller 自选举。

如何保证消息可靠性？

Kafka 只能对已提交的消息做有限度的持久化保证。

  1. 已提交：Kafka 的若干个 Broker 成功地接收到一条消息并写入到日志文件
  2. 有限度的保证：至少有一个 Broker 存活。

消息的发送是通过网络传输的，使用 APM 链路跟踪，从记录中找出哪个环节丢失的，重新处理

  1. 生产者：
    • 使用 producer.send(msg, callback) 接受回调事件
    • 设置 retries > 0，如果生产者发送 Broker 时出现网络抖动，Broker 没接收到，通过回调事件重试
    • 设置 acks = all，需要接收到 min.insync.replicas 的 ack 确认，才能确认消息已成功提交
  2. Broker：
    • 设置 replication.factor >= 3，指副本个数，至少 3 个以上，保持消息冗余
    • 设置 min.insync.replicas > 1，控制的是消息至少要被写入到多少个副本才算是已提交，但是也要保证 Broker 数量不能低于这个配置，即 replication.factor > min.insync.replicas
    • 关闭 unclean.leader.election.enable = false，避免一个 Broker 落后原先的 Leader 太多，当它成为 Leader 时必然会造成消息丢失
  3. 消费者：
    • 关闭 enable.auto.commit = false 自动提交，由程序手动提交，如果提交失败，再写入数据库，留到下一次重试。
    • 设置 auto.offset.reset=early 代替 lastet，新建分区时，Producer 先于 Consumer 发送消息，Consumer 能从最早的 Offset 消费，避免消息丢失。

如何保证消息幂等性？

如果设置了 retries > 0，Kafka 遇到网络抖动未写入 Broker、ISR 没有全部 ack 或者消费者没有提交 Offset 等情况，就会根据配置的次数重发消息，这意味着消费者会收到重复的消息。解决这个问题的办法有：

  1. 使用分布式锁来控制并发竞争资源，如果遇到网络问题，业务处理的时间超过了分布式锁的超时时间，这个时候业务通过重试机制又发送了同样的消息，此时分布式锁已经失效了，幂等性问题还是存在，所以还要其他方案辅助。
  2. 使用状态机，例如，订单状态可分为待支付、已支付、支付成功、支付失败，状态只能往后更新
  3. 使用数据库的唯一约束作为幂等性的最后保障，主要是考虑如果是分库分表的情况，将唯一键生成策略带上分库分表标志位就可以了。

如何保证消息顺序性？

根据 Kafka 顺序写入的特点，可以保证 Topic 同一个分区下消息的顺序性

  1. 单分区：不需要处理，但是高并发下可能会积压消息
  2. 多分区：利用 Procuder 选取分区的策略来实现，取出业务的共性字段来路由到同一个分区，有 2 种做法：
    • 通过程序计算，例如，取模，哈希，根据计算的分区索引直接指定分区
    • 设置为消息的 key，让 Procuder 根据 key 进行 hash 到同一个分区

Kafka 支持的消息机制？

  1. 至少一次：Kafka 默认提供的机制
  2. 最多一次：将重试机制 retires 关闭，会导致消息可能丢失
  3. 精确一次：需要 Kafka 的 版本为 0.11+
    • 幂等性 Producer，只能保证单分区上的幂等性
    • 事务型 Producer，保证多分区以及多会话上的消息无重复，需要开启幂等性，同时将 isolation.level 从 read_uncommitted 默认值改为 read_committed

消费者组是怎么重平衡的？

Kafka 为消费者组定义了一组状态机：

  1. empty：无消费者，有成员加入时会进入 prepare rebalance
  2. dead：消费者组过期
  3. prepare rebalance：预重平衡，完成重平衡进去 complete rebalance，当发生组成员加入或者离开时触发
  4. complete rebalance：完成重平衡，等待分配，分配完之后进入 stable
  5. stable：已分配，当组成员心跳失效或者离开会转到 prepare rebalance

重平衡主要在消费端和 Broker 端发生：

  1. 消费端：
    • JOINGROUP 过程：假设成员 A 是已加入的成员，现有成员 B 加入，向协调者发送 JOINGROUP 请求，协调者将成员 A 是领导者（这里跟 Leader/Follower 选举无关，指第一个加入的成员），由它分配消费分区，成员 A 也向协调者发送 JOINGROUP 请求，此时协调者回应成员 B 加入了，要消费哪个主题
    • SYNCGROUP 过程：接着成员 A 向协调者发送 SYNCGROUP 请求，将成员 B 的分配信息发过去，这个时候成员 B 也向协调者发送 SYNCGROUP 请求，协调者回应成员 B 消费哪个主题的分区，进入 stable 状态
  2. Broker 端：
    • 新成员加入组
    • 组成员主动离组
    • 组成员崩溃离组
    • 组成员提交位移

如何避免消费者组重平衡？

消费者组发生重平衡的触发条件有：

  1. 组成员数量发生变化
  2. 当订阅的主题包含通配符时，主题数量发生变化
  3. 分区数发生变化

由上面可以知道，主题和分区数可以通过早期的系统分析来解决。我们主要从组成员这方面来避免：

  1. 组成员如果奔溃了，可能会发生重平衡。我的想法是，组成员是通过心跳机制来维持存活状态，可以使用一些服务监控软件，如果发现服务进程奔溃了，然后马上重启，服务启动的时间能控制在 session.timeout.ms 之内完成一次心跳请求，就能避免重平衡，所以，服务要轻量级，秒级启动。
  2. 前期确定好组成员，不要有频繁的组成员加入。

为什么 Kafka Streams 很少人用？

  1. 应用部署依赖开发人员去做
  2. 目前只支持和 Kafka 集成，不像 Flink 可以接入 MySQL、HBase
  3. 依赖 Kafka 集群的协调功能来提供高容错性和伸缩性。

消费者如何根据偏移量找到消息？

消息存储在 Kafka/logs/主题+分区 目录下的 .log 文件，同目录下还有 .index 文件指向 .log。每个 log 文件按段（Segement）分割成多个文件，按最小的 Offset 命名，每间隔一个段的大小创建一个新文件， 段的大小通过 log.segment.bytes 指定，顺序读写，如下：

如上图，假设分段目录下有个 00000000000000368796.index 文件，Consumer 消费一个 Offset 为 368801 的 message 流程：

  1. 从分段的目录通过二分查找获取 .index 和 .log 文件，因为没有 368801 的文件，那么取 368796 的索引文件；
  2. 由于 Kafka 使用稀疏索引（如上图，1、4、7 之间跳过了部分索引），368801 相对于 368796 的相对索引号为 5，因此通过二分查找获取到相对 Offset 为 4 的位置，从中获取实际的物理位置 256；
  3. 从 368796 的物理文件直接找到物理位置 256 的消息，根据消息的 Offset 顺序往下查找，最后找到 368801 的消息。

Kafka 如何管理 TCP 连接？

  1. Producer 创建 TCP 连接
    • KafkaProducer 实例创建时启动 Sender 线程，从而创建与 bootstrap.servers 中所有 Broker 的 TCP 连接。注意： bootstrap.servers 一般配置 3~4 个即可，每个连接可以获取到集群下所有的 broker 元数据信息。
    • KafkaProducer 实例首次更新元数据信息之后，还会再次创建与集群中所有 Broker 的 TCP 连接。
    • KafkaProducer 发送消息到某台 Broker 时发现没有与该 Broker 的 TCP 连接，也会立即创建连接。
  2. Producer 关闭 TCP 连接
    • 在 Producer 端设置 connections.max.idle.ms > 0，TCP 连接会到期关闭，如果为 -1，不会关闭连接，变成僵尸链接
  3. Consumer 创建 TCP 连接
    • KafkaConsumer 在实例创建时不会创建 TCP 连接，而是 poll() 拉取数据时，向 Kafka 集群发送 FindCoordinator 获取当前消费者组的协调者，这是创建请求的 TCP 连接
    • 当 Broker 处理了消费者发来的 FindCoordinator 请求后，关闭创建请求的 TCP 连接，然后与协调者建立 TCP 连接
    • 消费者根据消费的分区所在的 Broker 建立 TCP 连接，例如，一个 Topic 的分区数为 5，这些分区所在的 broker 数为 4，那么创建 4 个 TCP 连接。
  4. Consumer 关闭 TCP 连接
    • 在 Consumer 端设置 connections.max.idle.ms > 0，TCP 连接会到期关闭，如果为 -1，不会关闭连接，变成僵尸链接

Consumer 获取消息是单线程吗？

Consumer 是双线程的设计，在获取消息为单线程：

  1. 用户主线程：负责获取消息，使用单线程轮询方式容易实现异步非阻塞，便于消费者扩展成支持实时流处理。
  2. 心跳线程：负责向 Kafka 汇报消费者存活情况，避免消息处理速度慢被踢下线。

Kafka 的请求是怎么处理的？

Kafka 使用 Reactor 模式来处理请求。Reactor 模式是事件驱动架构的一种实现方式，特别适合应用于处理多个客户端并发向服务器端发送请求的场景。Kafka 的 Broker 端有个 SocketServer 组件（类似于 Reactor 模式中的 Dispatcher 请求分发器），由 Acceptor 负责分发请求到网络线程池，线程池的大小由 num.network.threads 确定，默认为 3。

接着，网络线程池将用户的请求放入一个共享请求队列，由 I/O 线程池从这个队列读取，执行真正的处理，I/O 线程池的大小由 num.io.threads 决定，默认为 8。 I/O 线程池处理完成后通过 Purgatory 组件（缓存延迟请求，例如配置了 ack=-1，就等待所有 ISR 确认后才返回）返回对应的请求。

总结：Client 请求 -> Kafka Broker SocketServer -> Acceptor 请求分发器 -> 网络 Pool -> 共享 Queue <- I/O Pool 读取并处理，通过 Purgatory 组件响应请求。

Apache Kafka 定义了多达 45 种请求协议，用于实现各种各样的交互操作。例如，PRODUCE 请求用于生产消息、FETCH 请求用于消费消息、METADATA 请求用于获取 Kafka 集群元数据信息，所有的请求都是通过 TCP 以 Socket 方式进行通讯的。

如何理解 Kafka 的副本机制？

Kafka 的副本指的是同一个分区下的所有副本保存有相同的消息序列（数据冗余），避免某个节点宕机丢失数据。副本分为 Leader 副本和 Follower 副本，Leader 副本提供 Producer/Consumer 的读写操作，而 Follower 唯一的作用就是通过异步的方式同步 Leader 副本。

Kafka 根据 replication.factor 配置的副本个数来管理副本，通过 Broker 的 replica.lag.time.max.ms=n 来设置 Follower 副本落后 Leader 副本的时间不能超过 n 秒。超过这个值的副本将从 ISR 集合移除，ISR 表示没有明显落后 Leader 的副本，作用有 2 个：Producer 端使用 ack=-1 时需要所有 ISR 确认才能算提交（防止消息丢失）， 还有万一 Leader 宕机，Kafka 通过 Zookeeper 对 ISR 的副本重新选举，由于很落后的副本已从 ISR 移除，不会造成消息丢失（如果开启 unclean.leader.election.enable 会让 ISR 之外的副本参与选举，这样会丢失消息）。

如何理解 Leader Epoch 机制？

Leader Epoch 在 Kafka 0.11 版本引入，用来解决高水位更新错配导致不一致的问题。

首先，高水位是用来定义消息的可见性，帮助 Kafka 王城副本的同步。下图中，已提交消息用来保障 Kafka 的持久化，消费者只能消费已提交消息，即图中位移小于 8 的所有消息。高水位 HW 在 8 这个位置，不能被消费者消费。LEO 在 15 这个位置，表示副本写入下一条消息的位移值，也就是说 HW 和 LEO 都属于未提交消息。

高水位的更新机制。下图中，Broker 0 上保存了某分区的 Leader 副本和所有 Follower 副本的 LEO 值，而 Broker 1 上仅仅保存了该分区的某个 Follower 副本。

如果没有引入 Leader Epoch 机制。下图，副本 B 重启后，执行日志截断操作，将 LEO 值调整为之前的高水位值 1，删掉位移值为 1 的消息。当执行完截断操作后，副本 B 开始从 A 拉取消息，这个时候副本 A 宕机了，副本 B 被选举为 Leader。副本 A 恢复后，执行相同的日志截断操作，恢复和副本 B 一样的高水位，可以发现，位移值为 1 的消息就丢失了。

为了解决这个问题，需要引入 Leader Epoch，如下图的 Epoch 和 Offset 构成。 副本 B 重启后，**先从副本 A 获取 Leader 的 LEO，发现 LEO 的值不比自身的 LEO 小，不进行截断。**然后，A 宕机了，B 成为 Leader。接着 A 恢复了，执行相同的操作，从 B 获取 LEO 是未截断的，所以也不需要截断，位移值为 1 的消息在两个副本中均得到保留。

Kafka 为什么不支持读写分离？

Kafka 在 2.4 后提供了有限度的读写分离，就是说 Follower 副本能够对外提供服务，但是读写分离的实现没有必要：

  1. 读写分离适用于读多写少的场景，明显消息队列是不适合。
  2. 需要处理 Follower 滞后的问题。

Kafka 在哪个场景使用零拷贝？

  1. index 文件使用 mmap，通过 MappedByteBuffer 让用户态和内核态共享内核态的数据缓冲区，数据不需要复制到用户态空间。由于在不同的操作系统，mmap 的创建和销毁成本不一样，所以 Kafka 只有索引应用了 mmap 机制。
  2. logs 文件使用 NIO 包的 Channel.transferTo() 方法。

Kafka 实战

如何保证 Kafka 高可用？

  1. 配置 Broker 集群
  2. 配置连接的 Zookeeper，注意使用 chroot 作为别名
  3. 配置多个副本，让分区的副本分布到各个集群节点
  4. 配置 log.dirs 代替 log.dir 指向多块磁盘，支持同步读写多块磁盘和故障转移

统计生产端发送消费端的时长

Kafka 提供了拦截器的功能：

  1. 在生产端使用 ProducerInterceptor.send()
  2. 在消费端使用 ConsumerInterceptor.consume()
