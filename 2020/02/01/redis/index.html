Redis 基础

Redis 单线程为什么这么快？

Redis 在网络框架是单线程，使用多路复用，在 Linux 上使用 epoll 函数，因为 Redis 大部分的操作都是在内存上，使用多线程会产生上下文切换的开销，这个成本远大于单线程内存操作的时间。另外，Redis 其他操作如持久化、异步删除、集群数据同步是多线程的，这些功能并不要实时性，所以通过 fork() 子线程避免阻塞 Redis 主线程。

Redis 有哪些数据类型？

Redis 本身使用一个 RedisObject 的键值对（Hash）来保存所有管理的 key-value，value 可以是下面的数据类型。

Redis 常用的数据类型：

  1. Strings（字符串）：使用 sds
  2. Lists（列表）：数据量小用 ziplist（Redis 3.2）/quicklist（新版本），否则用 adlist
  3. Hashes（散列）：数据量小用 ziplist，否则用 dict
  4. Sets（集合）：数据量小用 intset，否则用 dict
  5. ZSets（有序集合）：数据量小用 ziplist，否则用 skiplist

内存阈值较低时，使用 ziplist 或者 intset 可以更省内存，本质上是数组，也就是 Redis 申请了连续的内存空间，充分利用了 CPU 的 cache line 特性（预读），对性能有很大的提升。

Redis 扩展的数据类型：

  1. Bitmaps（位图）：使用 n 个 hash 函数计算每个元素，将计算的 hash 值使用 64 位的二进制位。适合去重
  2. HyperLogLogs（统计）：用于基数统计，无论数据多大，只占 14 KB 左右。有一定的误差，一般用于统计月活、日活量
  3. Streams（消息队列）：

  2. 计数器：使用 String 的 set 实现
  3. 分布式锁：使用 String 的 set key value nx 实现，以前可以使用 lua 脚本来保证 setnx 和 expire 原子性操作，现在已经被新版本的 set 的 EX 和 NX 代替。
  4. 消息发布订阅：以前使用 Pub/Sub 订阅发布模式实现 m 对 n 的消费，允许 m 个发布者和 n 个订阅者。 相对专业的 Kafka 消息队列来说，缺少了消息持久化、Consumer Group 机制和 Offset 偏移消费的功能。所以 Redis 在后面又扩展了 Streams 数据类型。

  8. 查找附近的出租车：使用 Geo 类型

如何实现 Redis 事务？

Redis 通过 MULTI、EXEC、WATCH 等命令来实现事务功能。先以 MULTI 开始一个事务，然后将多个命令放入队列缓存中，最后由 EXEC 命令触发事务。Redis 的事务只是一个打包的批量执行脚本，事务过程中任意命令执行失败，不影响后续执行。

Redis 原理

Redis 用到哪些数据结构？

  1. 简单动态字符串：sds.c， 当保存的数据为 64 位有符号整数时，String 类型使用 int 编码，把它保存为一个 8 字节的 Long 类型整数，如果保存的是字符型，大小小于 44 byte，使用 embstr 编码（SDS、指针和数据在一块连续的内存区域），超过 44 byte 使用 raw 编码（指针和数据一起，SDS 分配独立的空间）。
  2. 双向链表：adlist.h
  3. 哈希字典：dict.h
  4. 整型集合：intset.h
  5. 压缩列表：ziplist.h
    压缩列表没有数据结构代码定义，完全是通过内存的特殊编码方式实现的一种紧凑存储数据结构，主要包含 zlbytes、zltail 列表尾的偏移量、zllen 列表长度和后面的 zlentry，内存布局为 zlbytes+zltail+zllen+{zlentry, zlentry ... ...}+zlend，如下所示： 目前，数据类型 Hash 和 List、ZSet 使用到了压缩列表，如果为了节省内存占用，可以修改 ziplist 的阈值`hash-max-ziplist-entries` 最大元素个数和 `hash-max-ziplist-value` 单个元素的最大长度。使用 ziplist，查询的复杂度为 $O(n)$，所以 ziplist 的阈值不能太大，需要权衡。另外，list 上使用 ziplist，增删数据还会产生连锁更新的问题。
  6. 跳跃表：skiplist.h

  7. 基数统计：hyperloglog.h
  8. RDB 快照：侧重于快速恢复，默认开启，在生成快照期间，为了避免有 Redis 请求进来导致这些请求没有包含在快照文件，Redis 使用操作系统的写时复制来保证期间的请求复制到快照中。触发快照的方式有执行 save() （阻塞主线程）、 bgsave() 命令， shutdown 优雅关闭 Redis、flushdb/flushall 清空缓存命令，或者按下面的配置指定周期触发：

AOF 将每条 Redis 的操作指令在文件尾部顺序写入，当 Redis 重启后根据 aof 文件逐条执行，意味着文件会越来越大，恢复数据的时间更长，当超过配置的阈值，先生成 temp-rewriteaof-[pid].aof 临时文件，然后替换原来的 AOF 文件，一旦发生重写，会造成未执行的数据丢失。Redis 生成的 aof 文件是可以直接打开查看的，从内容中可以看出 aof 主要存取 Redis 命令和相关参数信息，单个指令的大小要比原来的还大，所以文件较大时，需要对每个指令进行重放，这样恢复 Redis 的速度很慢。

RDB 相对 AOF 来说，通过快照的方式恢复更快，但是生成快照的时机一般是每秒执行一次快照，在这一秒内如果发生 Redis 奔溃重启，就会丢失数据。需要注意的是，RDB 快照过程中如果发生读写怎么办？Redis 在 RDB 快照过程中使用 fork 的 Copy On Write 写时复制操作，避免发生写入操作时导致数据不一致，由主进程 fork 出子进程，子进程需要拷贝进程必要的数据结构，如内存页表（虚拟内存和物理内存的映射索引表，如果将 save 频率设置得更快，就会造成 Redis 卡顿）并指向与父进程相同的内存地址空间，主线程修改原数据对子进程可见。被修改的数据会被复制一份为副本（会导致内存占用 2 份），bgsave 把副本数据写入 rdb 文件。

所以，最好的办法是 AOF + RDB 组合来保证数据不丢失，从 Redis 4.0 版本开始就提供了这样的机制，RDB 快照以一定的频率执行，在两次快照之间，使用 AOF 日志记录这期间的所有命令操作，恢复数据时，执行 RDB 快照（很快，但不全）+ AOF 追加（恢复上一个快照和当前快照之前的命令操作，不会太大，恢复快）。

一个很经典的例子，2 核 CPU、4GB 内存、500G 磁盘，Redis 实例占用 2GB，写读比例为 8:2，此时做 RDB 持久化会有什么风险？

  1. 从内存来看：在 4GB 的内存空间，Redis 的实例 占了一半，剩下 2G。在执行 RDB 持久化时，根据 COW 写时复制的特性，根据写比例为 8 可以知道 Redis 复制了 2GB * 0.8 = 1.6G，那么剩下的内存不到 400MB，操作系统本身也要内存吧，如果后续还有新的 Redis 键值请求，那么 Redis 有两种情况，第一种是 Linux 开启了 SWAP 机制，当内存不够用时，Redis 的部分数据被换入磁盘，性能急剧下降，第二种是没有开启 SWAP 机制，Redis 的进程直接 OOM 奔溃。
  2. 从 CPU 来看：CPU 是双核的，根据 Redis 的工作原理，主线程负责命令操作，Redis 可能会创建 RDB 的子线程和 AOF 的子线程和其他程序竞争最后一个线程，导致 Redis 的性能很差。如果通过 taskset 执行了绑核操作，将 Redis进程绑定到一个特定的 CPU，那么 Redis 下的所有子线程会互相竞争这个 CPU，那就完蛋了。所以，如果 Redis 要同时开启 RDB 和 AOF，不能进行绑核操作和开启 Huge Page。

主从同步如何实现一致性？

假设有 3 个 Redis 实例，Master（172.16.0.1:6379）、Slave1（172.16.0.2:6379）、Slave2（172.16.0.3:6379）

  1. 第一阶段：在 Slave1 执行 slaveof 172.16.0.1 6379 命令，向 Master 发起 psync ? -1，其中，psync 包含 RunID 和 Offset，RunID 指的是每个 Redis 实例启动时都会自动生成的一个随机 ID，Offset 表示同步的偏移量，首次同步设置为 -1，建立连接，告诉 Master 即将进行同步，Master 收到 psync 命令时，回复 FULLRESYNC {RunID} {Offset} 响应命令，带上 Master 的 RunID 和 Offset，Slave1 保存这些信息。
  2. 第二阶段：首次同步 Master 向 Slave1 发送 RDB 文件，Slave1 收到文件后，清空自己现有的数据和 RDB 信息，执行快照恢复
  3. 第三阶段：接下来 Master 向 Slave1 发送 replication buffer，记录第二阶段生成 RDB 文件之后的所有 Redis 写操作。

为了避免一个 Master 向多个 Slave 发送同步信息，Redis 使用级联的方式来传递同步信息，如上面的过程，Slave2 可以加入 Slave1，成为 Slave1 的从节点，也就是 Master -> Slave1 -> Slave2。

如果主库和从库发生网络中断，Redis 2.8 版本以前从库和主库会重新进行一次全量复制，开销很大，从 Redis 2.8 版本之后使用增量复制。主库会把断连期间收到的写操作命令写入 repl_backlog_buffer 环形缓冲区，记作 master_repl_offset，从库记录自己复制的偏移量，记作 slave_repl_offset。主从库的连接恢复之后，从库首先会给主库发送 psync 命令，并把自己当前的 slave_repl_offset 发给主库，主库会判断自己的 master_repl_offset 和 slave_repl_offset 之间的未同步的命令操作，通过 replication buffer 发送给从库。

repl_backlog_buffer 环形缓冲区是循环写入的，会覆盖，如果发生覆盖，Redis 会执行全量同步，可以把 repl-backlog-size 设大。

Redis 有哪些数据淘汰策略？

  1. volatile-lru：从已设置过期时间的数据集(server.db[i].expires)中挑选最近最久未使用的数据来淘汰。
  2. volatile-lfu：从已设置过期时间的数据集(server.db[i].expires)中挑选最不经常使用的数据淘汰。
  3. volatile-ttl：从已设置过期时间的数据集(server.db[i].expires)中挑选将要过期的数据淘汰。
  4. volatile-random：从已设置过期时间的数据集(server.db[i].expires)中任意选择数据淘汰。
  5. allkeys-lru：从数据集(server.db[i].dict)中挑选最近最少使用的数据淘汰。
  6. allkeys-lfu：从数据集(server.db[i].dict)中挑选最不经常使用的数据淘汰。
  7. allkeys-random：从数据集(server.db[i].dict)中任意选择数据淘汰。
  8. no-eviction：禁止驱逐数据，一般不推荐使用。

Redis 实战

如何搭建 Redis Cluster？

Redis Cluster 是 Redis 官方去中心化的集群方案，总共分配 13484 个虚拟槽位，通过 CRC16(key) % 13484 计算出 key 存储的目标槽位，用于解决单机内存不足的问题。使用主从复制模型，每个主节点 master 应至少有一个从节点 slave，搭建一个集群至少包含 6 个节点，3 主 3 从，其中主节点提供读写操作，从节点作为备用节点，不提供请求，只作为故障转移使用。

哈希槽底层使用 slots_to_keys 跳跃表记录了槽和键之间的关系，score 是槽号，member 是 key。

由于 Redis 实例之间会把自己的哈希槽信息发给其他连接的实例，所以客户端可以通过一个实例获得所有的哈希槽信息。但是，Redis 实例的增减会导致哈希槽重新分配，客户端无法感知这边变化，Redis Cluster 提供了一种重定向机制。当客户端把一个键值对的操作请求发给一个实例时，如果这个实例上并没有这个键值对映射的哈希槽，那么，这个实例就会给客户端返回下面的 MOVED 命令响应结果，这个结果包含了新实例的访问地址。

那么，配置集群的方式有 3 种：手动 meet、ruby 脚本、redis-cli –cluster

  1. 手动配置
    任选某个节点的客户端连接其他节点 分配哈希槽 slot，使用 cluster info 查看当前集群状态，由于哈希槽 slot 未分配，集群处于下线状态 cluster_state:fail 使用 cluster addslots 分配哈希槽 slot 使用 cluster replicate 设置从节点
  2. 如果 Redis 是 3.X 和 4.X 的版本，使用 redis-trib.rb 来配置集群。
    首先安装 Ruby 语言运行环境，版本在 2.5 以上，接着安装 Redis 的 Ruby 驱动 redis.gem 然后使用 redis-trib.rb 创建集群 创建完成后使用 redis-cli 验证
  3. 如果 Redis 是 5.X 以上的版本，使用 redis-cli 配置集群

如何搭建 Redis Sentinel？

Redis Sentinel 哨兵机制用于解决主从库故障转移的问题。使用一个运行在哨兵模式下的 Redis 进程，负责对 Redis 主从库发送 PING 命令来检测是否在线，如果主库没有在规定的时间内返回，就判断下线，根据一定的规则来选主，选主完成后把新主库的连接信息发给其他从库，让它们执行 replicaof 命令。

判断下线的方式需要部署 2N+1 个哨兵，因为通过一个哨兵来判断是不准确的，有可能是网络延迟导致。所以通过每个哨兵判定主观下线，然后将判定的结果，确定下线的判定结果是否超过 2N+1 的一半来决定是否下线，即客观下线。

选主的规则分 4 轮，如果在第一轮选主完成就不用进入下一轮：

  1. 通过 down-after-milliseconds 判断从库断开主库的超时次数来打分，如果断开次数越多分数越低
  2. 通过 slave-priority 给不同的从库设置不同优先级，那么优先级高的会被选为主库
  3. 通过主从复制的 master_repl_offset 和 slave_repl_offset 判断哪个从库最接近主库
  4. Redis 每个实例还有一个 ID，ID 越小，得分越高。

哨兵实例之间的发现，是通过 Pub/Sub 发布订阅来获取到双方的 IP 地址和端口的。在主从集群中，主库上有一个名为 __sentinel__:hello 的频道。在下图中，哨兵 1 把自己的 IP 和端口发布到 __sentinel__:hello 频道上，哨兵 2 和 3 订阅了该频道就可以从这个频道直接获取哨兵 1 的 IP 地址和端口号。

哨兵除了彼此之间建立起连接形成集群外，还需要和从库建立连接。在哨兵的监控任务中，它需要对主从库都进行心跳判断，而且在主从库切换完成后，它还需要通知从库，让它们和新主库进行同步。所以，哨兵向主库发送 INFO 命令获取从库的 IP 地址和端口，从而连接每个从库。
