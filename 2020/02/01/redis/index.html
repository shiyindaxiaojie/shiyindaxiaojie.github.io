Redis 基础

Redis 有哪些数据类型？场景？

Redis 本身使用一个 RedisObject 的键值对（Hash）来保存所有管理的 key-value，value 可以是下面的数据类型：

  1. Strings（字符串）：使用 sds
  2. Lists（列表）：数据量小用 ziplist（Redis 3.2）/quicklist（新版本），否则用 adlist
  3. Hashes（散列）：数据量小用 ziplist，否则用 dict
  4. Sets（集合）：数据量小用 intset，否则用 dict
  5. ZSets（有序集合）：数据量小用 ziplist，否则用 skiplist，可用于排名、高频搜索词
  6. Bitmaps（位图）：使用 n 个 hash 函数计算每个元素，将计算的 hash 值使用 64 位的二进制位。适合去重
  7. HyperLogLogs（统计）：用于基数统计，无论数据多大，只占 14 KB 左右。有一定的误差，一般用于统计月活、日活量
  8. Streams（消息队列）：待研究

Redis 原理

Redis 用到哪些数据结构？

  1. 简单动态字符串：sds.c
  2. 双向链表：adlist.h
  3. 哈希字典：dict.h
  4. 整型集合：intset.h
  5. 压缩列表：ziplist.h
    压缩列表没有数据结构代码定义，完全是通过内存的特殊编码方式实现的一种紧凑存储数据结构，内存布局为 zlbytes+zltail+zllen+{zlentry, zlentry ... ...}+zlend，一个压缩列表可以包含任意多个节点，每个节点可以保存一个字节数组或者一个整数值。

  7. 基数统计：hyperloglog.h
  8. RDB 快照：侧重于快速恢复，默认开启，在生成快照期间，为了避免有 Redis 请求进来导致这些请求没有包含在快照文件，Redis 使用操作系统的写时复制来保证期间的请求复制到快照中。触发快照的方式有执行 save() （阻塞主线程）、 bgsave() 命令， shutdown 优雅关闭 Redis、flushdb/flushall 清空缓存命令，或者按下面的配置指定周期触发：

AOF 将每条 Redis 的操作指令在文件尾部顺序写入，当 Redis 重启后根据 aof 文件逐条执行，意味着文件会越来越大，恢复数据的时间更长，当超过配置的阈值，先生成 temp-rewriteaof-[pid].aof 临时文件，然后替换原来的 AOF 文件，一旦发生重写，会造成未执行的数据丢失。Redis 生成的 aof 文件是可以直接打开查看的，从内容中可以看出 aof 主要存取 Redis 命令和相关参数信息，单个指令的大小要比原来的还大，所以文件较大时，恢复 Redis 的速度很慢。

RDB 相对 AOF 来说，通过快照的方式恢复更快，但是生成快照的时机一般是每秒执行一次快照，在这一秒内如果发生 Redis 奔溃重启，就会丢失数据，如果将 save 频率设置得更快，就会造成 Redis 卡顿，因为 主线程通过 fork() 创建子线程，这个创建的过程是阻塞主线程的。

所以，最好的办法是 AOF + RDB 组合来保证数据不丢失，从 Redis 4.0 版本开始就提供了这样的机制，RDB 快照以一定的频率执行，在两次快照之间，使用 AOF 日志记录这期间的所有命令操作，恢复数据时，执行 RDB 快照（很快，但不全）+ AOF 追加（恢复上一个快照和当前快照之前的命令操作，不会太大，恢复快）。

一个很经典的例子，2 核 CPU、4GB 内存、500G 磁盘，Redis 实例占用 2GB，写读比例为 8:2，此时做 RDB 持久化会有什么风险？

  1. 从内存来看：在 4GB 的内存空间，Redis 的实例 占了一半，剩下 2G。在执行 RDB 持久化时，根据写时复制的特性，根据写比例为 8 可以知道 Redis 复制了 2GB * 0.8 = 1.6G，那么剩下的内存不到 400MB，操作系统本身也要内存吧，如果后续还有新的 Redis 键值请求，那么 Redis 有两种情况，第一种是 Linux 开启了 SWAP 机制，当内存不够用时，Redis 的部分数据被换入磁盘，性能急剧下降，第二种是没有开启 SWAP 机制，Redis 的进程直接 OOM 奔溃。
  2. 从 CPU 来看：CPU 是双核的，根据 Redis 的工作原理，主线程负责命令操作，Redis 可能会创建 RDB 的子线程和 AOF 的子线程和其他程序竞争最后一个线程，导致 Redis 的性能很差。如果通过 taskset 执行了绑核操作，将 Redis进程绑定到一个特定的 CPU，那么 Redis 下的所有子线程会互相竞争这个 CPU，那就完蛋了，如果 Redis 要同时开启 RDB 和 AOF，不能进行绑核操作。

主从同步如何实现一致性？

假设有 3 个 Redis 实例，Master（172.16.0.1:6379）、Slave1（172.16.0.2:6379）、Slave2（172.16.0.3:6379）

  1. 第一阶段：在 Slave1 执行 slaveof 172.16.0.1 6379 命令，向 Master 发起 psync ? -1，其中，psync 包含 RunID 和 Offset，RunID 指的是每个 Redis 实例启动时都会自动生成的一个随机 ID，Offset 表示同步的偏移量，首次同步设置为 -1，建立连接，告诉 Master 即将进行同步，Master 收到 psync 命令时，回复 FULLRESYNC {RunID} {Offset} 响应命令，带上 Master 的 RunID 和 Offset，Slave1 保存这些信息。
  2. 第二阶段：首次同步 Master 向 Slave1 发送 RDB 文件，Slave1 收到文件后，清空自己现有的数据和 RDB 信息，执行快照恢复
  3. 第三阶段：接下来 Master 向 Slave1 发送 replication buffer，记录第二阶段生成 RDB 文件之后的所有 Redis 写操作。

为了避免一个 Master 向多个 Slave 发送同步信息（带宽 + CPU 消耗），Redis 使用级联的方式来传递同步信息，如上面的过程，Slave2 可以加入 Slave1，成为 Slave1 的从节点，也就是 Master -> Slave1 -> Slave2。

当主从库断连后，主库会把断连期间收到的写操作命令，写入 replication buffer，同时也会把这些操作命令也写入 repl_backlog_buffer 这个环形缓冲区（循环写入，会覆盖，需要把 repl-backlog-size 设大，如果发生覆盖，Redis 会执行全量同步），而从库写入 slave_repl_offset。主从库的连接恢复之后，从库首先会给主库发送 psync 命令，并把自己当前的 slave_repl_offset 发给主库，主库会判断自己的 master_repl_offset 和 slave_repl_offset 之间的未同步的命令操作，发送给从库。
