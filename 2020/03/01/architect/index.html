架构视野

一个系统从无到有，会经历什么架构历程？

  1. 单体架构：业务优先，初创公司
  2. 分布式架构：性能优先，传统软件公司/互联网
  3. SOA 架构：企业集成为主，服务治理为辅，中大型传统软件公司
  4. 微服务架构：服务治理，伸缩性，DevOps，互联网/大型公司

如何理解 CAP 定理？

CAP 定理指的是一致性、可用性和分区容错性三者只能同时满足两个。一致性指的是分布式系统同一时刻数据保持一致，可用性指的是使用集群保证每个请求都能得到正确的响应，分区容错性指的是网络发生故障导致各分区无法通信。分区容错性属于网络硬件问题，如果网路/路由器发生故障，导致多地的机房无法通信，只能 GG 了，例如支付宝遇到光纤被挖断导致几个小时无法使用，所以我们只能假设恒成立，那么我们只能在一致性和可用性权衡。

如何选择缓存模式和组件？

缓存模式有三种：

  1. Cache Aside (旁路缓存)：写操作 -> 先更新 DB，后清除 Cache，读操作 -> 先读 Cache，Cache MISS 就读 DB，再更新到 Cache。适用于强一致性场景。
  2. Read/Write Through（读写穿透）：写操作 -> 如果 Cache 不存在就只更新 DB，否则更新 DB + Cache，读操作 -> 先读 Cache，如果 Cache 不存在，就查 DB 再更新到 Cache 返回。适用于冷热数据场景，例如微博的最新用户列表，当用户发微博时，直接更新 DB，有 Cache 就顺带更新。
  3. Write Behind Caching（缓存触发异步写）：写操作 -> 先更新缓存，然后异步更新 DB，读操作 -> 先读 Cache，如果 Cache 不存在，就查 DB 再更新到 Cache 返回。适用于写入性能要求高但一致性要求不高的场景，会有丢失数据的风险。

架构要素

系统架构的要素有哪些？

  1. 高性能：分布式、消息队列、缓存、算法
  2. 高可用：集群、负载均衡、主从复制、保证 99.99+ 可用
  3. 扩展性：设计模式、分层架构、微服务
  4. 伸缩性：微服务、Docker
  5. 安全性：HTTPS、漏洞攻击

互联网常常提到的高并发是一种场景，高并发的最基本要求是高性能（用户体验）+ 高可用（零故障），所谓的三高（高并发、高可用、高性能）在理解上很容易误导。

实现高性能的方法有哪些？

提升系统的性能，无非就是从 CPU、内存、磁盘、 网络这些方面开展，根据木桶原理，磁盘和网络往往是性能的瓶颈，目前所有的优化无非就是空间换时间，如缓存、预读、缓冲区，基本上都是使用内存来避免 I/O 的开销，要么就是压缩数据，减少数据的查找次数。

  1. 优化 CPU 运算次数

    方法 说明
    使用最优的算法 一般来说，查找用数组，插入删除用链表，全文搜索使用 Trie 树。
    读多写少就预存 例如 Redis 大部分数据结构，都是写入的时候先计算好长度，读取的时候复杂度为 $O(1)$，当然，Redis 在处理命令是单线程的内存操作，可以很快地计算出长度。
    预测容量的大小 预先计算一些数据结构的最大容量，避免不断扩容，损耗 CPU，如 Java 的 ArrayList。
    优化锁的性能影响 优先无锁化，乐观锁（版本控制），CAS（非短时作业不要使用自旋锁），锁的升级，锁粒度。
    利用 CPU 原生指令集运算 使用位运算操作二进制。
    多核下使用多线程 使用线程池管理线程，并发运行。
    使用协程避免内核切换 例如 Go 的实现，在用户态进行上下文切换，比线程更轻量。

  2. 减少磁盘 I/O 次数

    方法 说明
    使用缓存来减少磁盘 I/O 次数 空间换时间，如本地缓存、Redis 缓存。
    使用合理的数据结构减少 I/O 次数 例如 MySQL 使用 B+ 树代替红黑树，降低树的高度（高度一般为 3），从而使得磁盘 I/O 次数也就为 3。
    利用索引覆盖高频数据 来自于 MySQL 的覆盖索引，如果索引页的节点足够返回所需数据，那么不需要回表查找。
    对数据进行预读 例如 MySQL 根据 extents 的读取量（阈值），预读取下一个 extents（一个区对应 64 页）。
    写入磁盘前先写入缓冲池 例如 MySQL 每次事务写入数据，先写入缓冲，然后根据一定的频率再刷回磁盘。
    使用顺序写的方式在尾部追加记录 避免磁道寻找次数，如 Kafka 将消息队列顺序写入 Logs 文件。
    使用内存映射操作磁盘文件 将磁盘映射到内存来操作，然后在适当的时机刷回磁盘，例如 Kafka 使用 mmap 读写文件。
    避免内核态和用户态的上下文切换和传输 例如 Kafka 和 Netty 都用到了零拷贝技术。
    文件较多时使用二分查找定位 例如 Kafka 写入的消息时根据 Offset 分段存储，十几亿的消息意味着大量的 log 文件，因为 Kakfa 是按照 Offset 顺序创建文件的，所以可以通过二分查找快速定位文件。

  3. 减少网络请求次数和大小

    方法 说明
    使用数据压缩提高吞吐量 HTTP 通过 gzip 减少网络传输的带宽占用。
    简化查询参数 为了避免浏览器的字符限制，可以看下 Google 将查询参数简化成什么样了。
    使用业务聚合代替不必要的 RPC 调用 一个 TCP 过程太多工作，本地调用完全可以忽略这些性能损耗，针对高并发场景可以这样极端处理。
    合并 HTTP 请求，减少握手次数 例如，合并多个图片一次请求下载
    多客户端使用 IO 多路复用模型 Linux 的 epoll 函数和 kqueue 函数。
    使用更高版本的 HTTP 协议 一般推荐 HTTP2，做了头部压缩，支持握手时发起多次请求，HTTP 3 使用 QUIC 和 UDP 代替了 TCP，应该会更快，但是还没普及。

  4. 其它

    方法 说明
    读写分离 读多写少，特指 MySQL 主从复制
    负载均衡 将服务器压力均摊到不同的机器，例如，设计一个支撑 100W 并发秒级写入的方案，一台 Redis 实例预计每秒 81000 条，根据数据结构折后估计能达到 50000 条，那么可能需要部署 20 个实例。
    分区/分表/分库 例如 MySQL 单表超过 2000W 时，树的高度会严重影响查询性能，先分表，但数据库本身的 I/O 频率较高时，再分库
    使用列式存储 海量数据使用 HBase，HBase 在存储上每个列是顺序存储的，意味着磁盘转动次数更少

如果说让系统更快是为了拼用户体验，抢占市场，那么让系统节省空间就是为了成本开支，用节省下来的钱做更多的投资。一般是从内存、磁盘来考虑，内存一般是避免 OOM 奔溃，磁盘就是别写满了。

  1. 优化内存占用

    方法 说明
    使用循环代替递归 在一个方法不停的递归，每个方法产生一个栈帧，直到栈空间不够，抛出 SOF 错误。有些情况下，例如斐波那契数列，使用递归还会导致多余的运算。
    使用 Bitmap 优化大数据量的去重 使用布隆过滤器防止缓存穿透，缺点是需要消耗 CPU 来计算多个哈希函数。
    优化大量重复前缀的 key 例如 Redis 的 key，有些人使用某些特定的业务前缀存取了十几亿的 Key，导致内存占用很大，将这些 key 压缩为 Set 或者 Hash，可以节省掉这个业务前缀占用的空间。

  2. 减少磁盘占用

    方法 说明
    使用更小字节的数据类型创建索引 能用数值就尽量用数值，如果必须用字符型，那么就看下有没有区分度，找下这类字符有没有重复的地方，把这段重复的字符截断，减少索引的占用。
    使用稀疏索引 这个来自于 Kafka 定位消息 Offset 的场景，如果每条消息都建立索引来对应物理位置，空间占用就会很大，使用稀疏索引 + 二分查找，时间复杂度为 $O(logn)$，再加上历史的消息文件定期会被清除，所以性能上是可以接受的。
    正确清理文件的策略 一般使用 Kafka 和 Elasticsearch 这些大数据场景的中间件，需要根据磁盘空间调整下清理策略，否则，遇到高并发场景或者压测，磁盘很快就满了，程序也跑不起来。还有，使用 Logback 这类日志框架

实现高可用的方法有哪些？

  1. 负载均衡：选择适当的路由算法，例如，轮询、权重、IP、Hash，类似的工具有 OpenResty（Nginx + Lua）、Haproxy、Httpd，当然，性能最好的是 F5 硬件，就是成本太高。
  2. 数据分片：主要是 NoSQL 和消息队列，例如，Redis Cluster 使用 16384 槽位分片 + 副本，Kafka 基于 Zookeeper 的 ZAB 协议（应该是参考 Paxos 选举算法实现的）实现多分区 + 副本。
  3. 服务治理：双向注册，心跳检测，例如，Eureka 使用一个注册器管理所有的服务（内存操作），Eureka 自身也是可以互相注册的，没有单点故障的问题。

架构模式

系统有哪些架构模式？

  1. 分层：MVC、三层架构、六边形架构
  2. 分割：业务拆分、数据库拆分
  3. 集群：Nginx、Haproxy、Httpd
  4. 分布式：微服务、Docker、Kibernates
  5. 缓存：Redis、Memcached
  6. 异步：Kafka、RocketMQ
  7. 冗余：数据冗余、配置冗余、备份恢复
  8. 自动化：Jenkins + Maven/Gradle + Git
  9. 安全：OAuth2、JWT、HTTPS、RSA

怎么理解 MVC 和三层架构的关系？

MVC（View -> Controller -> Model）是 Spring Framework 的设计框架，核心是通过 DispatcherServlet 请求分发，是 Web 和后端交互的一种规范，三层架构（Web -> Service -> DAO）根据职责划分层次，是通用的设计思想，MVC 在这个思想强化了 Web 层的工作，增加了路由和视图、拦截器这些新特性，这两者的关系为 Web 包含 View + Controller，Model 包含 Service + DAO。

架构设计

如何设计分布式全局唯一性的 ID？

  1. 数据库自增：适用于一主多从，如果多个数据库自增，会出现重复，另外，根据数据量来选择 int 还是 bigint。
  2. 雪花算法：Twitter 开源的，使用 64 位数字确定 ID，结构为 0 + 41 位时间戳 + 10 位机器 ID + 12 位序列号，确定唯一性的关键地方是通过 synchronized 加锁，然后对比上一个时间戳和当前时间戳，如果上一个时间戳比当前时间戳还大，说明系统的时间被修改，或者 Linux 执行 ntpdate 同步时间有问题，这个时候一直循环获取当前系统的时间戳，直到比上一个时间戳大才能往下执行。ShardingSphere 提供了 Java 版本的实现SnowflakeShardingKeyGenerator。
  3. 美团点评生成 ID 解决方案：美团点评的 Leaf 框架提供了 LeafSegement 号段、LeafSnowflake 组合 Zookeeper 。LeafSegement 就是预先分配 ID，假设步长为 1000，那么先执行第一次 SQL 查询先分配 1000 给业务模块在内存使用，当号段用完了就再触发一次 SQL 查询，分配下一个号段。LeafSnowflake 在雪花算法的基础上增加了 Zookeeper，用来获取和分配 workId 机器 ID。

如何确保分布式场景的并发幂等性？

  1. 前端幂等性：从业务源头控制，但是解决不了非页面的重复请求。

    • 按钮点击一次马上置灰不可用，然后倒计时。
    • 表单提交后重定向到结果页，避免用户刷新表单重复提交，例如京东、淘宝支付订单都是跳转到结果页的。

  2. 后端幂等性：GET 请求肯定是幂等的，这里是要解决 POST/PUT/DELETE 请求的幂等性问题。

    • insert 场景先使用 Redis 的 set 加锁，并设置失效时间，MySQL 在锁期间执行 select 判断是否存在，然后进行 insert，由于锁并发，第一个请求进入成功 insert 后，后续的请求可能因为获取不到锁一直重试，等到 Redis 的锁失效了，然后提交重复的请求，解决这个问题的办法是使用数据库唯一性约束作为后备校验，fang’zhi任意时刻同一个表单的重复提交。
    • update/delete 场景可以使用版本控制，也可以使用单向状态机实现幂等，例如，订单支付的状态有待支付 -> 支付中 -> 支付成功 -> 支付失败，更新语句为 update order set status = '支付成功' where status = '支付中'，在支付中的记录无论更新多少次，只能是支付成功或者支付失败。

如何设计高并发场景的限流策略？

  1. 前端控制：验证码，表单校验，对特殊场景定制业务限制，如秒杀活动需要报名
  2. 负载分流：集群下的负载均衡，将流量分摊到多个节点，如 Nginx/Openresty 负载均衡
  3. 服务限流：可以根据用户请求的 IP 或者特定的标识限流
    • 固定窗口算法：在固定的时间窗口按照阈值限流，例如，10 s 内设置阈值为 1000 条，当超过 1000 条就阻塞，缺点就是遇到突发流量时，第一次请求 1998 条，阻塞 998 条，等 10 s 过后创建一个新的时间窗口，阻塞的 998 条和后续的 2 条进入，可想而知，如果流量一直没降下来的话，后面会阻塞更多的请求。
    • 漏桶算法：定义一个有容量的桶，所有的请求都拥挤到一个漏口出去（限速），当桶满了就丢弃请求，无法满足突发流量。
    • 令牌桶算法：定义一个有容量的桶，将请求当作令牌，定期往桶里发放一定的令牌，遇到突发流量不会限速，当桶满了就丢弃令牌。

限流算法目前最好的就是令牌桶算法了，可以使用 Google Guava 的 RateLimiter 工具、Alibaba 开源的 Sentinel 框架，Netflix Eureka Client 也内置了 RateLimiter 。

如何保证缓存和数据库的一致性？

先更新数据库，然后清除缓存，延迟一秒后再清除一次缓存，假设线程 A 和 B 并发，更新的值 V = 0（缓存也为 0）：

  1. 线程 A 更新数据库设置 V = 1 -> 线程 B 读取缓存 V = 0 -> 线程 A 清除缓存 V -> 线程 A/B 读取缓存不存在，从数据库读取 V = 1 并设置到缓存。
  2. 线程 A 更新数据库设置 V = 1 -> 线程 A 清除缓存 V -> 线程 B 读取缓存不存在，从数据库读取 V = 1 并设置到缓存
  3. 线程 A 更新数据库设置 V = 1 -> 线程 A 清除缓存 V 失败，将失败操作写入 Kafka 再重试，或者通过 Alibaba 开源的 cannel 订阅 bin-log 来清除。
  4. 缓存 V 刚好失效，线程 B 读取缓存，缓存为空，从数据库取旧数据 -> 线程 A 更新数据库 -> 线程 A 清除缓存 -> 线程 B 更新旧数据到缓存 -> 这种情况线程 A 需要延迟一秒再删（双删）。

架构方案

怎么设计一个秒杀系统？

秒杀系统主要是考虑如何应对突发大流量的问题，我认为可以从业务、前端、后端这 3 个方面分析：

  1. 业务控制

    • 商品预热，报名了才能秒杀某些商品，不过淘宝没有这样做，只是通过聚划算去分流。
    • 下单预扣库存，付款释放库存，超时未付款时会自动释放库存，重新上架，淘宝的秒杀步骤有 3 步：购买/购物车 -> 提交订单（去库存） -> 支付。

  2. 前端优化

    • 按钮置灰，秒杀前几分钟，用户可能在不停的刷新，所以要做好页面的 CDN 缓存，整体只有一个倒计时在执行，时间到就打开按钮，按钮一旦点击就跳转到订单确认页面，淘宝好像是点击购买后（库存第一次校验），先跳转到提交订单（库存第二次校验），然后再跳转到支付页面。
    • 使用验证码或者问答题防止机器人和非法用户请求。

  3. 后台负载

    • 令牌桶限流：使用限流过滤器应对突发流量，如 Google Guava 的 RateLimiter 工具、Alibaba 开源的 Sentinel 框架。
    • 服务降级：秒杀期间，降级非关键业务服务，减少对秒杀业务的影响，腾出空间扩容秒杀业务。
    • 服务熔断：对于超时请求直接快速失败或者转移到后备服务，避免占用大量的 TCP 连接。
    • 缓存热点数据：二八原则，可以根据区域、类型、用户画像划分数据区域。
    • 消息队列缓冲：缓解数据库的压力，异步消费，在下单预扣库存的场景，使用时延队列回收超时未支付的订单。
    • 代码简化：合并 RPC 调用减少跨服务请求的开销，简化 Spring MVC 的请求代码

短链接是怎么设计的？

短链接是什么？淘宝一键分享，让对方复制后打开淘宝就可以跳转到对应的商品，或者运营商/银行推送短信，后面带了一个短链接，点击后跳转到相应的长链接页面。

为什么要用短链接？

  1. 字数限制，如微博、朋友圈、短信平台
  2. 通过短链接生成的二维码更容易识别
  3. 某些平台无法识别原生链接的一些参数

怎么实现短链接？

  1. 使用 Murmur Hash 非加密哈希算法将长链接生成短链接，例如，将某些原生链接生成 http://gk.link/a/1234567890， 还可以进一步缩短长度，将 1234567890 十进制转化为 62 进制，缩短到 6 位，例如，http://gk.link/a/1ab2cd
  2. 为了避免哈希碰撞，使用短链接和长连接的关系表维护，在插入关系前通过布隆过滤器判断是否存在，如果不存在就插入。读的时候直接通过短链接取出长链接。
  3. 基于 HTTP 协议，在 HTTP Status Code 返回 302 临时重定向，从 Response Headers 的 Location 输出长链接，点击短链接时会跳转过去。

为什么不是用 MD5/SHA1 哈希算法？因为短链接没有加密的必要，只要能区分原始链接就可以了，Murmur 非加密哈希算法的性能更高。
