Spring 基础

Spring 事务机制

Spring 在 spring-framework-tx 包提供了事务管理的接口，如平台事务管理 PlatformTransactionManager、事务定义 TransactionDefinition、事务运行状态 TransactionStatus。Spring 本身不直接管理事务，而是提供了多种事务管理器，将事务管理的职责委托给 Hibernate 或者 JTA 等持久化机制所提供的相关平台框架的事务来实现。

Spring 事务管理接口 PlatformTransactionManager 定义了如何获取事务运行状态、对事务运行状态进行提交或者回滚的方法。

PlatformTransactionManager 接口实现 说明 所属类库
CciLocalTransactionManager ~ spring-tx
ChainedTransactionManager ~ spring-data-commons
DataSourceTransactionManager JDBC 事务管理器，JdbcTemplate 和 Mybatis 可以共用 spring-jdbc
HibernateTransactionManager Hibernate 事务管理器，分为 3/4/5 版本 spring-orm
JpaTransactionManager JPA 事务管理器 spring-orm
JtaTransactionManager JTA 事务管理器 spring-tx
Neo4jTransactionManager Neo4j 事务管理器，整合 JPA 事务需要解决冲突 spring-data-neo4j
PseudoTransactionManager ~ spring-integration-core
RabbitTransactionManager RabbitMq 事务管理 spring-rabbit
ResourcelessTransactionManager ~ spring-batch-infrastructure
WebLogicJtaTransactionManager WebLogic 事务管理器 spring-tx
WebSphereUowTransactionManager WebSphere 事务管理器 spring-tx

Spring 事务定义接口主要设置了事务的规则。

Spring 事务运行状态接口用于获取事务的执行状态。

Spring 允许使用声明式事务 @Transactional 注解到类或者方法中。

Spring 事务的执行过程：

  1. 在方法 A 标记注解后，Spring 通过代理在调用方法之前开启事务，在调用方法返回之后执行提交事务，调用方法抛出异常时回滚事务；
  2. 如果方法 A 调用另外一个被 Spring 管理的类的方法 B，方法 B 也标记了注解，根据方法 B 的注解判断传播行为，例如按默认的行为 Propagation.REQUIRED，那么，由于方法 A 开启了事务，方法 B 也加入（共享）了这个事务，正常情况下，方法 B 如果抛出异常，方法 A 和 方法 B 都回滚事务；
  3. 如果方法 B 设置了 Propagation.REQUIRED_NEW，从方法 A 调用 B 时，B 会新建一个事务，并挂起（暂停）A 的事务，B 执行完后，A 恢复事务，不管 B 是否回滚都不影响 A 的事务；
  4. 如果方法 C 和方法 A 都属于同一个类，方法 A 直接用函数调用内部方法 C，C 的事务不起作用，原因是基于 AOP 代理，切面只能在 Spring 接口生成的代理类的方法生成事务，解决这个问题，可以分离方法 C 到另外一个类 C，在类 A 注入类 C 进行调用。

Spring 事务的回滚机制：

  1. 异常被 try-catch 捕获异常，但没有抛出异常，不回滚；
  2. 定义的方法不是 public，不回滚；
  3. 在接口标记注解，由于 AOP 代理机制只能作用在代理类，不回滚；
  4. 父类的类上标记注解，子类所有方法只要出错都会回滚；
  5. 父类的方法上标记注解，且子类重写了该方法一旦出错也会回滚，但是这个时候子类其他方法调用此方法，出错不会回滚；
  6. 基于 AOP 代理，事务只能作用在 Spring 管理的类；
  7. 回滚默认只接受运行时异常和 Error，如果需要受检异常有效，需要显示的标记为 @Transactional(rollbackFor=Exception.class)。

Spring 缓存机制

Spring 从 3.1 版本开始引入了基于注解的缓存接口，在 spring-context 包中定义了 Cache 和 CacheManager 两个接口来统一不同的缓存技术。Cache 接口包含缓存的常用操作，如增加、删除、读取等。CacheManager 是管理各种缓存技术的抽象接口。

Spring 的 CacheManager 接口的实现类如下：

CacheManager 接口实现 Cache 接口实现 说明
SimpleCacheManager 无 使用 java.util.Collection 存储缓存
NoOpCacheManager NoOpCache 测试时使用，不会实际存储缓存
ConcurrentMapCacheManager ConcurrentMapCache 使用 java.util.ConcurrentHashMap 存储缓存
CompositeCacheManager 无 组合多个 CacheManager，通过轮询得到相应的缓存
EhCacheCacheManger EhCacheCache 使用 EhCache 存储缓存，是 Hibernate 中默认的缓存方案
JCacheCacheManager JCacheCache 使用 JCache（JSR-107）标准的缓存技术
GuavaCacheManager GuavaCache 使用 Guava 存储缓存
CaffeineCacheManager CaffeineCache 使用 Caffeine 存储缓存，代替 Guava 缓存技术
TransactionAwareCacheManagerProxy TransactionAwareCacheDecorator 基于 Spring 事务代理缓存
RedisCacheManager RedisCache 使用 Redis 存储缓存
HazelcastInstanceConfiguration HazelcastCache 使用 Hazelcast 存储缓存

Spring 声明式的缓存注解来操作缓存。

实现类 说明
@Cacheable 在方法或者类上注解，通过 value 指定缓存名称，调用该方法时先查询缓存，缓存如果存在，直接返回，不继续调用该方法，否则调用该方法，将返回值放入缓存
@CacheEvict 在方法或者类上注解，通过 value 指定缓存名称，用于从缓存中清除过期、指定或未使用的数据
@CachePut 在方法或者类上注解，通过 value 指定缓存名称，每次调用该方法不做缓存检查，都把返回值更新到缓存
@Caching 在方法或类上注解，允许组合多个缓存操作以应用于方法
@CacheConfig 只能在类上注解，当一个类中有很多个缓存操作是重复的，使用这个配置解决
@EnableCaching 开启缓存配置

Spring 原理

如何理解 Spring IOC ？

Spring IOC 指的是控制反转，将实例化的工作托管到 Spring 容器来完成，实现控制反转有 2 种方式：

  1. 依赖注入 DI：例如 @Autowired、构造函数、BeanFactoryRegister、XML装配
  2. 依赖查找 DF：ApplicationContext.getBean

Spring 使用三级缓存解决了单例的循环依赖的问题（A -> B，B -> C，C -> A），非单例需要通过 @Lazy 组合 AOP 生成代理来解决。

Spring IOC 最核心的接口是 BeanFactory，IOC 的配置工作将 Bean 注册到 BeanDefinition 的 Map 中，整个过程涉及到相关的重要组件有：

  1. DefaultListableBeanFactory：默认的 BeanFactory 实现，完全实现了所有 BeanFactory 接口；
  2. BeanDefinition: 定义了 Bean 的配置信息，如 isLazyInit 是否来加载、是否单例（singleton/prototype）等；
  3. BeanDefinitionRegistry：定义 Bean 的注册接口，由 DefaultListableBeanFactory 实现将 Bean 注册到 beanDefinitionMap 中。

Spring IOC 的配置分为 XML 文件配置和 Java 注解配置。XML 文件配置的入口为 ClassPathXmlApplicationContext。

Spring MVC 是怎么处理请求的？

  1. 以 DispatcherServlet.doService() -> doDispatch() 为入口
  2. 根据 request 请求从 HandlerMapping 获取对应的 HandlerExecuteChain
  3. HandlerExecuteChain 包含 HandlerObject（注册的 @Controller）和拦截器
  4. 在 doDispatch() 方法中，先执行拦截器的前置方法，一般在这里可以设置黑白名单、认证提前返回
  5. 然后通过 HandlerAdapter 适配器包装 HandlerObject 执行请求，返回统一的对象 ModelAndView，交由 ResolveView 处理
  6. 最后执行拦截器的后置方法

Spring MVC 接收客户端请求的核心是 DispatcherServlet，其初始化涉及到 3 个重要组件，针对我们编写的 handler object（@Controller/@RestController）进行处理：

  1. HandlerMapping：配置 request 与 handler object（或者 HandlerExecutionChain）的映射，负责匹配请求的URL、方法、参数、Content Type等；
  2. HandlerExecutionChain：包装 handler object 和 interceptor 拦截器的执行链。
  3. HandlerAdapter：handler object 的适配器，将请求参数转换为 handler 能接受的数据类型，解析参数、处理返回值等。

Spring 拦截器和过滤器有什么区别？

过滤器是 J2EE 规范的定义，依赖 Servlet 容器并由其创建，可以在源头过滤所有的 Servlet 请求，而拦截器是 Spring 自定义的接口，在过滤器之后执行，本质上是在 Servlet 内实现细粒度的方法拦截（return 掉），如果过滤器对 HTTP 请求做了处理，有可能进不到拦截器。

Spring 拦截器在 DispatcherServlet 的 handler 处理方法前后设置了前置拦截和后置拦截，如果我们在实现 Interceptor 接口返回了 false，就不会进入到 handler 处理步骤，如下：

Spring 实现 AOP 有 2 个方案：

如何理解 MyBatis 缓存机制?

MyBatis 分为一级缓存和二级缓存：

  1. 一级缓存使用 Executor 内部的 PerpetualCache 存取查询结果，PerpetualCache 就是用一个普通的 HashMap 来存，由于一级缓存只在单个 SqlSession 中，也就是和 SqlSession 共享生命周期。 如果遇到多个 SqlSession 操作同一张表，会有脏数据的问题，一般情况下，建议关闭一级缓存。
  2. 二级缓存解决多个 SqlSession 共享的问题 二级缓存开启需要 3 个条件:
    • 在 mybatis-config.xml 总配置文件设置 cacheEnabled 属性为 true
    • select 语句所在的 Mapper 配置 <cache/> 或者 <cache-ref/> 节点
    • select 语句的参数加上 useCache 属性为 true 二级缓存每个 Cache 都有自己的 Namespace，可以通过 <cache-ref namespace=""/> 指定相同的命名空间实现多个 Mapper 共享同一份缓存。
