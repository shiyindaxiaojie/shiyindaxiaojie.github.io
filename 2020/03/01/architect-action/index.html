动环监控系统

应用于数据中心的运维监控系统，对云计算基础设施的设备及环境进行管理、巡检、监控、分析，核心功能如下：

  1. 数据采集：实时同步第三方系统的设备的测点数据，数据的存储周期至少要 6 个月以上。
  2. 数据分析：数据同步过来了，根据业主提供的公式对数据计算和呈现，例如，汇总一个机房内所有设备的某项数值的总和或者平均值。
  3. 告警推送：从第三方采集的数据包含设备的异常数据，但业主不希望发生太多警报，需提取关键告警。例如，一个机房发生断电，我希望只推送一条机房断电的故障，断电后，这个机房内的其他设备也会产生大量的告警，而这些告警是因为机房断电产生的，并不是自身的故障。

系统分析

从需求中提取关键因素：

关键因素 说明
用户 一个数据中心园区的运维人员不超过 300 人，因为排班的关系，峰值不超过 30 人。
站点 站点就是一个区域，可以是一个园区，也可以是一个机房，通过层级关系表示，目前只需要做到地市级的规模。
设备 设备和站点是多对一的关系，设备可以有不同的类型、型号，但编码是唯一的。
测点 测点就是某项数值，如电压 220V，一个设备可以包含多个测点，举个例子，一个机柜有温度的测点，也有电压的测点。

根据性能指标进行规模估算，确定边界，计算需要配置多少资源才能满足项目的需求：

性能指标 要求 临界条件
用户并发数 至少支持 30 人的并发 $Tomcat 请求并发数 >= 用户并发数 * 并发时 HTTP 请求数 + 后台非业务 HTTP 请求数$
第三方数据同步 延迟不能超过 30 秒 -
查询响应时间 不能超过 3 秒 -
数据存储周期 至少保留 180 天 -

架构设计

技术栈主要由 Spring Boot、Tomcat、Redis、Kafka、MySQL 构成。

性能优化

本人使用 Kafka 作为同步协议，通过优化分区解决了消费积压和顺序消费问题，利用 Redis 缓存热点数据来加快业务处理，最后对 MySQL 的库表进行优化，将平均写入速度从原来的每秒 2000 条提升到 25000 条。
